# Bug Fix Skill — 系统化缺陷修复工作流

## 设计思路

### 为什么需要这个 Skill

传统的 bug 修复流程有四个常见痛点：
1. **分析不透彻** — 拿到 bug 描述就动手改，没做根因分析，改了表象漏了本质
2. **影响不可控** — 修复 A 模块时没考虑对 B/C/D 模块的连锁影响，引入回归 bug
3. **过程不可追溯** — 修复过程全在脑子里，事后说不清"为什么这么改"
4. **验证不闭环** — 改了代码就跑，没有系统化验证，靠运气

### 设计原则

| 原则 | 说明 |
|------|------|
| **先分析后动手** | Step 1-6 全部分析，Step 7 才开始改代码。分析不透彻不进编码 |
| **影响驱动修改** | 不是"哪里报错改哪里"，而是先画影响矩阵，再在矩阵约束下找最小修改集 |
| **全程留痕** | 每一步的输出都写入日志文件，可追溯、可复盘 |
| **人机协作** | 分析由 AI 驱动，决策由人确认。每步支持跳过/重来/修改 |
| **渐进式降级** | E2E 能用则用，不能用则降级人工验证或跳过，不阻塞流程 |
| **顿悟兜底** | 连续失败触发重新思考机制，打破思维定势 |

### 核心概念

**影响矩阵** — 是本 skill 的核心创新。它是一个二维表：

```
         | 模块A | 模块B | 模块C | 模块D
---------|-------|-------|-------|-------
修改点1  |  高   |  中   |  低   |  无
修改点2  |  无   |  高   |  高   |  低
```

行 = 每个修改位置，列 = 系统中所有功能模块，单元格 = 该修改对该模块的影响程度（高/中/低/无）。

修改方案必须在影响矩阵的约束下选择"最小影响路径"——优先选修改点少、影响面窄的方案。

**顿悟模式** — 当同一轮修复超过 3 次 E2E 失败时触发。行为：
1. 放弃之前的分析缓存，从头重读原始缺陷描述
2. 并行启动 3 个子 agent：**架构视角**（系统级依赖关系）、**代码视角**（实现细节与边界条件）、**测试视角**（测试用例覆盖盲区）
3. 综合三方结论，生成全新的根因分析和修改方案
4. 标记旧日志为"已废弃"，新分析写入新日志

**Patch 导出** — 修复完成后，不直接 git add，而是先把修改过的文件（保持目录结构）复制到 patch 目录，附带 `.patch-meta.json` 记录修改摘要。让你可以脱离 IDE 审查每一次修改。

### 适用范围

- 复杂 bug（跨模块、根因不明）
- 需要追溯修复过程的质量场景
- 团队协作修 bug（日志可共享）
- 简单 bug（拼写错误、一眼看出）可快速走完前几步

## 前置条件检查

启动 skill 后，先检查：

1. Git 仓库状态 — `git status`，确保工作区干净或有明确的变更上下文
2. Playwright 可用性 — 检查 `npx playwright --version` 是否可执行
3. 日志目录 — 确保 `.archbot/bugfix-logs/` 存在，不存在则创建
4. Patch 目录 — 确保 `.archbot/bugfix-patches/` 存在，不存在则创建

## 工作流

### Step 1 — 录入缺陷描述

**目标**: 获取完整的、结构化的缺陷信息。

**输入**: 用户提供的缺陷描述（可以是自然语言、截图描述、错误日志等）

**AI 动作**:
1. 如果描述不完整，引导用户补充以下信息：
   - **现象**: 发生了什么？预期 vs 实际
   - **触发条件**: 什么操作/输入/环境触发？
   - **影响范围**: 影响哪些功能/用户？
   - **复现步骤**: 如何稳定复现？
   - **环境信息**: 浏览器/OS/Tauri 版本/相关配置
2. 根据缺陷描述，提取一个**中文简述**（6-12字），例如：
   - "登录页面崩溃"、"文件树展开异常"、"HTTP请求超时"
3. 生成日志文件名：`【中文简述】_YYYY-MM-DD-HHmmss.md`

**输出**: 将结构化缺陷描述写入日志文件头部。

**交互**:
- 提示用户确认中文简述，可修改
- 信息不完整时逐项询问

**用户可输入「结束」跳过** → 直接用现有描述进入 Step 2

**日志格式**:
```markdown
# 缺陷修复日志: 【中文简述】
> 创建时间: YYYY-MM-DD HH:mm:ss
> 状态: 进行中

## 1. 缺陷描述
### 现象
...
### 触发条件
...
### 影响范围
...
### 复现步骤
...
### 环境信息
...
```

---

### Step 2 — 分析描述

**目标**: 将原始描述转化为结构化的技术分析，排除歧义，确定问题边界。

**AI 动作**:
1. 识别关键实体：涉及的文件、模块、函数、API 端点
2. 识别问题类型：逻辑错误 / 边界条件 / 竞态条件 / 资源泄漏 / 类型错误 / 配置错误 / 集成问题
3. 判断问题层级：前端 / 后端 / 数据层 / 通信层 / 配置层
4. 搜索相关代码上下文（使用 Grep/Glob 定位相关文件）
5. 如果 E2E 环境可用，尝试复现并截图记录

**输出**: 追加到日志文件。

**交互**:
- 分析结论有歧义时暂停确认
- 「结束」跳过

**日志格式**:
```markdown
## 2. 描述分析
### 关键实体
- 文件: ...
- 模块: ...
- 函数: ...
### 问题类型
- 主类型: ...
- 子类型: ...
### 问题层级
- 层级: ...
### 代码上下文
(相关代码片段)
### 复现结果 (如有环境)
(截图/日志)
```

---

### Step 3 — 根因分析

**目标**: 使用 5-Why 法逐层深挖，找到问题的根本原因。

**AI 动作**:
1. 从现象出发，连续追问 5 层"为什么":
   ```
   现象: 点击登录按钮后页面白屏
   Why 1: 因为 JS 抛出了未捕获的异常
   Why 2: 因为 API 返回的数据格式与预期不符
   Why 3: 因为后端重构了响应结构，前端未同步更新
   Why 4: 因为前后端接口约定文档未及时更新
   Why 5: 因为缺少接口契约测试（根因）
   ```
2. 将根因分类到：代码缺陷 / 设计缺陷 / 流程缺陷 / 环境配置 / 第三方依赖
3. 明确指出：是最近引入的（回归）还是历史遗留

**输出**: 追加到日志文件。

**交互**:
- 任何一层 Why 的推理不确定时暂停确认
- 「结束」跳过

**日志格式**:
```markdown
## 3. 根因分析
### 5-Why 分析
Why 1: ...
Why 2: ...
Why 3: ...
Why 4: ...
Why 5: ... ← 根因

### 根因分类
- 类别: [代码缺陷 | 设计缺陷 | 流程缺陷 | 环境配置 | 第三方依赖]
- 引入时机: [最近回归 | 历史遗留]

### 根因描述
(一句话总结)
```

---

### Step 4 — 影响矩阵

**目标**: 梳理系统中受根因影响的所有模块，评估修改的波及面。

**AI 动作**:
1. 列出项目中所有功能模块（从 `function-map.yml` 或代码结构提取）
2. 对每个模块评估影响程度：
   - **高**: 该模块直接依赖被修改的代码，修改后行为必然变化
   - **中**: 该模块间接依赖，或共享数据结构/工具函数
   - **低**: 该模块仅通过接口交互，接口不变则不影响
   - **无**: 完全独立
3. 根据初步分析，列出候选修改点
4. 生成影响矩阵表（行=候选修改点，列=功能模块）

**输出**: 追加到日志文件。

**交互**:
- 模块列表是否完整 → 确认
- 影响程度判断有争议 → 确认
- 「结束」跳过

**日志格式**:
```markdown
## 4. 影响矩阵

### 功能模块清单
| 序号 | 模块名 | 路径 | 说明 |
|------|--------|------|------|
| 1 | ... | ... | ... |

### 影响矩阵
| 修改点 \ 模块 | 模块A | 模块B | 模块C | ... |
|---------------|-------|-------|-------|-----|
| 修改点1       |  高   |  中   |  无   | ... |
| 修改点2       |  低   |  高   |  中   | ... |

### 最小影响路径分析
推荐修改方案: ...（选择影响面最窄的方式）
需特别关注的模块: ...（标注高影响模块）
```

---

### Step 5 — 定位修改点

**目标**: 在影响矩阵的约束下，精确到文件和函数级别定位修改位置，给出修改思路。

**AI 动作**:
1. 对每个候选修改点：
   - 精确定位到文件路径 + 行号范围
   - 指出涉及的函数/类/组件
   - 说明当前代码的问题
   - 提出修改方向（不是最终代码，是思路）
2. 再次与影响矩阵交叉验证：该修改会不会波及其他模块？
3. 如果有多个修改点，排序优先级（先修改核心问题，再修改衍生问题）

**输出**: 追加到日志文件。

**交互**:
- 每个修改点的定位是否准确 → 确认
- 「结束」跳过

**日志格式**:
```markdown
## 5. 修改点定位

### 修改点 1: [简短描述]
- **文件**: `src/xxx/xxx.vue`
- **行号**: L120-L145
- **函数/组件**: `handleLogin()`
- **当前问题**: ...
- **修改方向**: ...
- **影响矩阵复核**: 对模块B有"中"影响，需在模块B的 `foo()` 中同步调整调用方式

### 修改点 2: ...
### 修改优先级
1. 修改点1 (核心)
2. 修改点2 (衍生适配)
```

---

### Step 6 — 修改方案

**目标**: 给出具体的、可执行的修改方案（伪代码/设计层面），让用户评估。

**AI 动作**:
1. 将 Step 5 的修改方向具体化为修改方案
2. 描述每个文件的具体变更内容（不写完整代码，写变更概要）
3. 说明跨文件的修改顺序
4. 标注风险点：哪些修改有不确定性、需要特别关注

**输出**: 追加到日志文件，**摘要输出到对话供用户评估**。

**交互**:
- **这是最重要的确认节点，必须等用户明确同意后才进入 Step 7**
- 用户可以要求修改方案、调整修改方式
- 「结束」跳过确认直接进入编码

**日志格式**:
```markdown
## 6. 修改方案

### 修改顺序
1. 先修改 `xxx.rs` 的 `foo()` — 核心逻辑修正
2. 再修改 `xxx.vue` 的 `handleBar()` — 前端适配
3. 最后更新 `xxx.ts` 的类型定义 — 类型同步

### 变更概要
#### 文件: `src-tauri/src/xxx.rs`
- 修改 `foo()` (L120-L145)
  - 将 ... 改为 ...
  - 增加 ... 边界检查
- 新增辅助函数 `validateXxx()` (L145后)
  - 负责 ...

#### 文件: `src/components/Xxx.vue`
- 修改 `handleBar()` (L80-L95)
  - 适配新的 API 响应格式
  - 增加 loading 状态处理

### 风险点
| 风险 | 级别 | 应对 |
|------|------|------|
| 修改 foo() 可能影响模块B | 中 | 在模块B的调用处增加兼容处理 |
| 新响应格式可能有未覆盖的边界情况 | 低 | 增加 try-catch 兜底 |

### 用户评估
> [ ] 方案确认，可以开始编码
> [ ] 需要调整: ___
```

---

### Step 7 — 编码修复

**目标**: 按照确认的修改方案，逐个文件修改代码。

**AI 动作**:
1. 按修改顺序，逐个文件执行 Edit/Write
2. 每次修改后，检查 lint/type 错误
3. 所有修改完成后，运行项目的类型检查/编译
4. 记录实际修改内容

**输出**: 追加到日志文件。

**原则**:
- 最小改动原则 — 改的越少，风险越小
- 每次只改一个逻辑点，方便回溯
- 不改和缺陷无关的代码（不趁机重构）

**注意**: 此步骤用户不需要逐行确认（方案已确认），但遇到方案未覆盖的意外情况时暂停询问。

**日志格式**:
```markdown
## 7. 编码修复

### 实际修改记录
#### `src-tauri/src/xxx.rs`
- L120-L145: 重写 `foo()` 逻辑
  - 修改前: (关键旧代码)
  - 修改后: (关键新代码)
  - 变更原因: (对应方案中的哪一条)

### 编译/类型检查结果
- [ ] cargo check 通过
- [ ] vue-tsc 通过
```

---

### Step 8 — 导出 Patch

**目标**: 将修改过的文件（保持目录结构）复制到 patch 目录，附带元信息。

**AI 动作**:
1. 通过 `git diff --name-only` 获取所有修改文件
2. 在 `.archbot/bugfix-patches/【中文简述】_YYYY-MM-DD-HHmmss/` 下创建对应目录结构
3. 复制修改后的文件（保持相对路径）
4. 生成 `.patch-meta.json` 记录：
   - 缺陷简述
   - 修改时间
   - 修改文件列表（原始路径 → patch 内路径映射）
   - 每个文件的修改摘要

**输出**:
```
.archbot/bugfix-patches/登录崩溃_2026-05-30-143000/
├── src-tauri/
│   └── src/
│       └── auth.rs
├── src/
│   └── components/
│       └── Login.vue
└── .patch-meta.json
```

**日志追加**:
```markdown
## 8. Patch 导出
- **导出路径**: `.archbot/bugfix-patches/登录崩溃_2026-05-30-143000/`
- **包含文件**:
  - `src-tauri/src/auth.rs` — 修正 token 过期处理
  - `src/components/Login.vue` — 适配新的错误响应格式
```

### Step 9 — E2E 验证（可选）

**目标**: 如果运行环境可用，通过 Playwright 验证修复效果。

**环境检测**:
1. 检查 Playwright 是否可用：`npx playwright --version`
2. 检查是否有可运行的开发服务器
3. 检查是否有现成的 e2e 测试用例

**路径 A — Playwright 自动化（环境就绪）**:
1. 如果没有 e2e 目录，创建 `e2e/` 和基础配置
2. 编写针对该缺陷的回归测试用例
3. 启动开发服务器，运行测试
4. 记录测试结果

**路径 B — 人工验证（Playwright 不可用但有浏览器）**:
1. 启动开发服务器（如可行）
2. 引导用户在浏览器中按复现步骤验证
3. 询问用户验证结果（通过/不通过）

**路径 C — 跳过（无运行环境）**:
1. 记录"无可用的运行环境，跳过 E2E 验证"
2. 直接进入 Step 10

**失败处理**:

```
失败次数 = 0

WHILE (测试不通过):
    失败次数 += 1
    
    IF 失败次数 <= 3:
        1. 将测试失败信息追加到日志
        2. 回到 Step 2，根据失败信息重新分析
        3. 重新执行 Step 2-8
        4. 再次运行测试
    
    IF 失败次数 > 3:
        触发 → 顿悟模式
        BREAK
```

**顿悟模式**:
1. 在日志中标记：`> ⚠️ 已失败 3 次，进入顿悟模式`
2. 废弃之前的分析日志章节（追加 `[已废弃]` 标记）
3. 创建新日志文件：`【中文简述】_顿悟_YYYY-MM-DD-HHmmss.md`
4. 并行启动 3 个子 agent（Agent 工具），传入原始缺陷描述和所有历史日志：

   | 子 Agent | 视角 | 分析方法 |
   |----------|------|---------|
   | Architect | 架构视角 | 从系统依赖关系、模块边界、接口契约重新审视问题 |
   | Code Explorer | 代码视角 | 从实现细节、边界条件、副作用重新审视问题 |
   | Test Analyzer | 测试视角 | 从测试盲区、未覆盖路径、断言有效性重新审视问题 |

5. 综合三方结论，产生全新的根因分析和修改方案
6. 回到 Step 2 继续流程

**日志追加**:
```markdown
## 9. E2E 验证

### 环境检测
- Playwright: [可用 | 不可用]
- 开发服务器: [可启动 | 不可启动]
- 已有e2e用例: [有 | 无]

### 测试执行
- 路径: [A | B | C]
- 结果: [通过 | 不通过]

### 失败记录 (如有)
- 第 N 次失败
- 失败现象: ...
- 错误日志: ...

### 顿悟模式 (如触发)
> ⚠️ 已失败 3 次，进入顿悟模式
> 旧日志: xxxxx.md [已废弃]
> 新日志: xxxxx_顿悟_YYYY-MM-DD-HHmmss.md
```

### Step 10 — 根因分析报告

**目标**: 对复杂问题，编写一份独立的根因分析报告。

**判定标准** — 满足以下任一条件即为"复杂问题":
- 涉及 3 个及以上文件的修改
- 根因跨越 2 个及以上层级（如前端+后端）
- 5-Why 分析触及流程/设计缺陷
- 影响矩阵中有"高"影响超过 2 个模块
- 触发了顿悟模式

**AI 动作** (仅复杂问题):
1. 将 Step 2-6 的分析内容精炼为报告格式
2. 包含章节：问题概述 → 根因 → 影响分析 → 修复方案 → 经验教训 → 预防措施
3. 保存到 `.archbot/bugfix-reports/【中文简述】_YYYY-MM-DD-HHmmss.md`

**简单问题**: 在此步骤记录"该问题为简单问题，无需根因分析报告"。

**日志追加**:
```markdown
## 10. 根因分析报告
- 复杂度判定: [复杂 | 简单]
- 报告路径: `.archbot/bugfix-reports/xxx.md` (如有)
- 结论: (如简单问题，简述原因)
```

### Step 11 — Git Add

**目标**: 将修改暂存到 git，等待用户 commit。

**AI 动作**:
1. 运行 `git status` 列出所有变更
2. 逐文件 `git add <file>` 暂存（不 commit）
3. 再次 `git status` 确认

**原则**:
- **只 add，不 commit**（commit 由用户决定时机）
- 不 add 无关文件（`.patch-meta.json` 等）
- 不 add patch 目录和日志目录的内容（已在 `.gitignore` 中）

**日志追加**:
```markdown
## 11. Git Add
- 已暂存文件:
  - `src-tauri/src/xxx.rs`
  - `src/components/Xxx.vue`
- 未暂存的辅助文件 (patch/日志):
  - `.archbot/bugfix-patches/...`
  - `.archbot/bugfix-logs/...`
```

## 交互协议

每个 Step 结束后的标准交互模式：

```
📋 Step N 完成: [步骤名]

分析结果: (摘要)

继续? A) 继续下一步  B) 修改当前步骤  C) 查看完整日志

```

**特殊指令**:
| 输入 | 效果 |
|------|------|
| `结束` | 跳过当前步骤，保留已有输出进入下一步 |
| `重来` | 重新执行当前步骤 |
| `看日志` | 输出当前日志完整内容 |
| `改 [内容]` | 修改当前步骤的特定输出 |
