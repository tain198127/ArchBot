# 输入框约束设计维度

## 一、数据属性维度

**这个字段"是什么"**

- **数据类型** — 字符串 / 整数 / 小数 / 日期 / 时间 / 日期时间 / 布尔 / 枚举 / 文件 / JSON
- **数据精度** — 小数保留几位、时间精确到秒还是毫秒
- **数据长度** — 字符数限制（汉字算1还是2）、字节数限制（数据库层面）
- **值域范围** — 数值的最小/最大值、日期的起止范围
- **格式规则** — 正则匹配（邮箱、手机号、身份证、车牌…）

## 二、业务规则维度

**这个字段"在业务上怎么用"**

- **必要性** — 必填 / 选填 / 条件必填（某状态下必填）
- **唯一性** — 全局唯一 / 租户内唯一 / 与另一字段组合唯一
- **默认值** — 固定默认 / 动态默认（取当前时间、取当前用户、取上一单的值）
- **可编辑性** — 新增时可编辑 / 编辑时可编辑 / 审核后只读 / 永久只读
- **可见性** — 始终可见 / 条件可见 / 仅特定角色可见

## 三、字段关联维度

**这个字段"和其他字段的关系"**

- **联动赋值** — A 选了 X，B 自动填充 Y
- **联动显隐** — A 选了 X，B 才显示
- **联动校验** — A 填了，B 就变必填
- **大小比较** — 开始时间 ≤ 结束时间，最低价 ≤ 最高价
- **一致性约束** — 确认密码 = 密码，报销总额 = 明细金额之和
- **互斥约束** — 手机号和邮箱至少填一个

## 四、交互体验维度

**用户"怎么感知和使用"这个字段**

- **校验时机** — 输入时实时校验 / 失焦校验 / 提交时统一校验
- **错误提示** — 位置（字段下方 / tooltip / 顶部汇总）、文案风格（简洁 / 引导式）、多错误时的展示策略
- **输入辅助** — placeholder 引导文案、输入计数（x/200）、历史输入建议
- **自动格式化** — 手机号 136-0000-0000、银行卡每4位空格、金额千分位
- **控件选择** — 文本框 / 下拉单选/多选 / 日期选择器 / 数字步进器 / 开关 / 级联选择

## 五、安全合规维度

**这个字段"有哪些风险和合规要求"**

- **敏感信息** — 是否需要脱敏展示（手机号 136****0000）、是否需要加密存储
- **输入过滤** — XSS 过滤、SQL 注入防护、特殊字符处理
- **密码安全** — 复杂度要求、不能与历史密码重复、弱口令黑名单、有效期
- **合规要求** — 个人信息保护法、GDPR、等保、行业监管要求（金融/医疗）

## 六、国际化维度

**这个字段"在不同地区怎么表现"**

- **日期格式** — yyyy-MM-dd（中） vs MM/dd/yyyy（美） vs dd/MM/yyyy（欧）
- **数字格式** — 小数点 . vs , ，千分位分隔符
- **货币** — 符号位置（¥100 vs 100円）、小数位数（日元0位、人民币2位）
- **字符集** — 是否允许 CJK / 阿拉伯文 / 特殊字符
- **时区** — 存储 UTC 还是本地时间、展示时如何转换

## 七、状态与生命周期维度

**这个字段"在不同阶段怎么变化"**

- **状态** — 正常 / 禁用 / 只读 / 隐藏
- **状态流转** — 什么事件触发状态变化（提交后只读、审批后锁定）
- **修改追溯** — 是否需要记录修改历史、是否需要留痕
- **废弃策略** — 字段不再使用时，是隐藏还是标记废弃

## 八、技术约束维度

**这个字段"受什么技术限制"**

- **数据库约束** — varchar(N)、字段类型能否扩展、索引影响
- **接口约束** — 参数大小限制、编码方式
- **性能** — 大批量校验是否影响响应速度、唯一性校验的查询成本
- **兼容性** — 历史数据能否满足新约束、旧接口是否需要兼容
- **扩展性** — 枚举值未来是否会增加、长度上限是否预留空间
