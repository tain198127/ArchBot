# 统一方法论生成与执行框架 2.1：评估修正版闭环稿

## 0. 修正说明

本稿是在《统一方法论生成与执行框架 2.1：全量融合清稿版》的基础上，依据《统一方法论：概念、关系、层次、结构判断标准》和《统一方法论评估报告：概念、关系、层次、结构正确性审查》完成的修正版。

本次修正目标不是增加内容，而是关闭评估报告中的结构性问题：

```text
P0-1：横切增强能力与主干工程化概念混用
P0-2：workflow-state-machine.yml 引用了未定义 concept
P0-3：concept card 中关系方向存在错误风险
P1-1：owner 枚举不一致
P1-2：mutable_by_llm 语义过粗
P1-3：主干层级与横切能力之间缺少显式映射
P1-4：本质抽象关系类型与概念关系类型混用
P2-1：16 个工程化概念没有全部补齐概念卡
P2-2：30 个元能力缺少能力卡
P2-3：策略选择前后顺序需要拆分
P2-4：方法论进化缺少版本控制规则
```

修正后的核心变化：

```text
1. 新增 object_taxonomy，明确主干概念、横切运行时对象、认知型元能力、生命周期演化对象。
2. 将 workflow 中的 required_concepts、required_runtime_objects、required_context_objects 分开。
3. 所有关系统一使用 from / to / type / meaning，禁止隐式方向。
4. owner 改为 owners 数组，避免出现未定义枚举 human_runtime。
5. mutable_by_llm 改为 llm_permissions，区分候选生成与最终修改。
6. 新增 cross_cutting_mapping，明确横切能力跨哪些层。
7. 概念关系类型与本质关系类型分命名空间。
8. 补齐 16 个工程化概念的概念卡。
9. 补齐 30 个元能力的能力卡。
10. 将 strategy_selection 拆成 preliminary_strategy_selection 与 final_strategy_selection。
11. 新增 methodology_versioning，约束方法论演化。
```

最终定位：

> 统一方法论是一套把经验变成概念、把概念变成结构、把结构变成配置、把配置变成执行、把执行变成反馈、把反馈沉淀为新方法论的系统。

# 1. 总体原则

```text
物理上统一，逻辑上分层。
领域不拆包，通过 domain_profile 适配。
环境识别先于情景识别。
情景识别先于初步策略选择。
概念澄清先于元能力提炼。
最终策略选择必须在概念澄清和关系建模之后确认。
元能力进内核。
组合套路变解释器。
具体方法论进配置。
业务变化优先改配置。
执行语义变化才改代码。
大模型只生成候选内容。
机器负责校验、门禁、状态流转、文件写入和审计。
人负责确认目标、边界、领域规则、低置信度项和关键裁决。
没有证据的内容不能进入最终结论。
低置信度内容必须进入人工审查。
洞察不能直接成为最终结论。
知识迁移不能直接复制，必须说明差异和风险。
方法论更新必须人工批准。
blocker 门禁失败必须阻断流程。
所有危险动作必须经过 policy engine。
最终产物必须通过 review checklist。
历史经验必须被记录、反思、提炼、再沉淀。
```

# 2. 对象分类体系 object_taxonomy

统一方法论中的对象必须先分类，再进入 schema、workflow、policy、gate 或 review。

```yaml
object_taxonomy:
  core_engineering_concepts:
    - mission
    - scope
    - non_goals
    - required_inputs
    - optional_inputs
    - deliverables
    - coordinator
    - subagents
    - workflow
    - stage_gates
    - execution_rules
    - decision_rules
    - context_files
    - file_structure
    - review_checklist
    - fallback_rules

  cross_cutting_runtime_objects:
    - domain_fit_gate
    - domain_profile
    - environment_profile
    - situation
    - evidence_traceability
    - confidence_policy
    - execution_reliability_rules
    - methodology_versioning

  cognitive_meta_capabilities:
    - environment_recognition
    - situation_recognition
    - contextual_insight_generation
    - historical_reflection
    - pattern_induction
    - methodology_distillation
    - strategy_selection
    - graph_understanding
    - knowledge_transfer
    - strategic_reasoning
    - feedback_learning_loop
    - essence_abstraction_and_transfer

  lifecycle_evolution_objects:
    - insight_candidate
    - pattern_candidate
    - essence_pattern
    - analogy_transfer
    - knowledge_transfer_record
    - methodology_update_candidate
    - methodology_version_record
```

判断规则：

```text
core_engineering_concepts 进入 concept-layers.yml 和 engineering-concepts.yml。
cross_cutting_runtime_objects 进入 runtime schema，不进入 10 个概念层。
cognitive_meta_capabilities 进入 capability registry，不进入 16 个工程化概念。
lifecycle_evolution_objects 进入 context / audit / evolution，不作为主干概念。
```

这样修正后，概念、运行时对象、认知能力、演化对象不会互相污染。

## 4.1 domain_fit_gate

```yaml
domain_fit_gate:
  pass_when:
    - has_stable_inputs: true
    - has_clear_deliverables: true
```

# 8. 关系类型命名空间 relation-types.yml

概念关系与本质关系必须分开。

```yaml
relation_type_namespaces:
  concept_relation_types:
    - contains
    - depends_on
    - produces
    - validates
    - controls
    - references
    - blocks
    - resolves

  essence_relation_types:
    - parallel
    - sequence
    - causal
    - depends_on
    - reinforces
    - conflicts_with
    - blocks
    - leads_to
```

规则：

```text
concept_relation_types 用于方法论结构图。
essence_relation_types 用于现象网络、本质图和因果模式。
两类关系不能混用。
```

# 9. 概念关系 concept-relations.yml

```yaml

relations:

    from: "workflow"
    to: "required_inputs"

    enforcement: "runtime"

    namespace: "concept_relation_types"

    from: "execution_rules"
```

# 27. 最终定稿原则

```text
概念不清，不进入元能力提炼。
情景不清，不进入策略选择。
证据不足，不进入最终结论。
置信度不足，不跳过人工审查。
规则不可执行，不进入 policy。
流程不可状态机化，不进入 workflow。
门禁不可判定，不进入 stage_gates。
产物不可验收，不进入 deliverables。
知识不可声明差异和风险，不允许迁移。
本质模式没有反例和适用边界，不允许沉淀。
方法论更新未经人工批准，不允许生效。
```

最终一句话：

> **统一方法论修正版，是一套把经验变成概念、把概念变成结构、把结构变成配置、把配置变成执行、把执行变成反馈、把反馈沉淀为新方法论的闭环系统。**
