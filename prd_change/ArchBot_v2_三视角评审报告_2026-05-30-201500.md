### 🔴 P1. 首次启动体验未设计

用户双击 ArchBot.app → 首次启动 → 解压 4 个 Runtime .tar.gz（~260MB）→ 初始化隔离 HOME。这个过程可能需要 30-60 秒，用户看到什么？

**建议**: 增加首次启动引导流程（onboarding wizard）：
1. 显示 "正在准备 AI Runtime..." 进度条
2. 逐 Runtime 展示解压进度
3. 完成后展示 Runtime 版本清单
4. 首次启动后不再重复此流程

### 🟡 P2. Runtime 隔离状态对用户不可见

A 子系统有 toast 告警（权限被拒绝时），但 B 子系统的隔离状态（是否正在使用隔离 HOME、审计是否发现违规）对用户完全不可见。

**建议**: 在设置面板增加 "Runtime 状态" 页签：
- 当前 HOME 路径（隔离目录）
- 最近审计结果（违规数 / 总操作数）
- 各 Runtime 版本号

### 🟡 P3. 多项目切换未定义

用户从项目 A 切换到项目 B 时，Runtime 进程是否终止？隔离 HOME 是否复用？A 子系统说 "project.close → db_disconnect + store.closeProject()"，但 B 子系统没有定义 Runtime 进程的生命周期。

# 第二视角: 技术

> 评审人设: 技术负责人。关注实现可行性、安全完备性、性能、测试、错误处理。

## T1. 实现可行性

| 检查项 | 状态 | 风险 |
|--------|------|------|
| 编排引擎 (A) | ✅ | 已有 v1 基础，增量改动小 |
| 配置隔离 (B) | ⚠️ | Rust 侧全新代码，跨平台差异 |
| 预打包 (C) | ⚠️ | 依赖外部包源（npm/pypi/GitHub Releases）的可用性 |

### 🔴 T1. Windows `env_clear()` 风险

`cmd.env_clear()` 在 Windows 上会移除 `SystemRoot`、`TEMP`、`USERPROFILE` 等系统关键变量。部分程序（包括 Node.js）依赖 `SystemRoot` 来定位系统 DLL。当前代码只注入了 `PATH`、`LANG`、`TZ`。

**修复**: Windows 上增加最小系统变量：

```rust
#[cfg(target_os = "windows")]
{
    cmd.env("SystemRoot", "C:\\Windows");
    cmd.env("TEMP", &config.isolated_temp);
    cmd.env("TMP", &config.isolated_temp);
}
```

### 🔴 T2. Adapter 协议未定义

A 子系统通过 NativeMenuFactory 的 action 回调触发 ActionRegistry.execute()。B 子系统启动 Runtime 进程。C 子系统分发 Adapter 包。但 **Adapter 与 ArchBot 之间的 HTTP API 协议完全没有定义**。

架构结论 §7-8 定义了接口（`/v1/sessions`, `/v1/sessions/{id}/turns`, SSE events），但统一方案中所有三个子系统都引用了 Adapter，却没有引用或复述这份协议。这会导致实现时三个子系统对 Adapter 的假设不一致。

**修复**: 在统一方案中增加一节 "Adapter 协议契约"，引用架构结论 §7-8 的核心接口定义，并明确 A/B/C 各自依赖的 Adapter 接口子集。

### 🟡 T3. Hermes Python C 扩展的平台兼容性

C.3 已指出 macOS ARM64 ↔ x64 不能交叉编译。Windows ARM64（Snapdragon X 笔记本）上 Python C 扩展可能不兼容。需要在 `runtimes/manifest.yml` 中显式声明 "Windows ARM64: 不支持"。

### 🟡 T4. `@tauri-apps/api/menu` 动态 import 失败处理

`NativeMenuFactory` 中使用 `await import('@tauri-apps/api/menu')`。在浏览器模式下，这个 import 会失败（`@tauri-apps/api/menu` 依赖 Tauri WebView runtime）。虽然有 `isTauri()` 守卫，但如果 `isTauri()` 误判（例如 Tauri 环境但 menu API 不可用），需要更明确的错误处理链。

## T2. 安全完备性

| 检查项 | 状态 |
|--------|------|
| Backend 命令白名单 | ✅ |
| 五层纵深防御 | ✅ |
| SHA256 完整性校验 | ✅ |
| **预打包源的可信性** | ⚠️ `npm pack` / `pip download` / GitHub Releases 的 TLS 依赖 |

预打包脚本从 npm registry、PyPI、GitHub Releases 下载——这些来源的 TLS 证书链是唯一信任锚。如果 CI runner 的 CA 证书被污染，整个分发链被攻破。建议在 `verify-checksums.sh` 中增加 GPG 签名验证（如果上游提供）。

## T3. 性能

| 指标 | 估算 | 风险 |
|------|------|------|

| ContextMenuResolver 索引构建 | <5ms | 30+ 菜单项，O(1) |
| ExpressionEvaluator 递归下降解析 | <1ms | L1 语法简单 |
| `env_clear()` + 重新注入 | <1ms | 操作系统开销 |

# 第三视角: 架构

> 评审人设: 架构师。关注子系统边界、耦合度、演进性、与现有架构的一致性。

## A1. 子系统耦合分析

```
A (编排引擎) ──────── 依赖 ────────> B (配置隔离)
     │                                 │
     │  通过 ActionRegistry.execute()   │  通过 RuntimeLauncher
     │  触发 backend action             │  启动进程
     │                                 │
     └──────── 共享依赖 ───────────────┘
                  │
         同一份 runtimes.yml 配置
         同一套 permissions.yml 规则
```

| 耦合点 | 类型 | 风险 |
|--------|------|------|
| `runtimes.yml` 被 B 和 C 同时引用 | 配置耦合 | 中等 — B 关注 isolation 段，C 关注 platforms 段，改一个可能影响另一个 |
| `permissions.yml` 被 A 使用，B 的审计规则独立定义 | 语义重复 | 低 — A 的 allowActions 和 B 的 forbidden_prefix 是不同维度 |
| A 的 ActionRegistry 触发 B 的 Runtime Launcher | 运行时耦合 | 低 — 通过 ActionRuntime.invoke() 解耦 |

### 🟡 A1. `runtimes.yml` 职责过载

当前 `~/.archbot/config/runtimes.yml` 同时承载：
1. B 子系统: `execution.isolation` 配置（隔离模式、HOME 路径、白名单）
2. C 子系统: 版本号、下载源（已被 `runtimes/manifest.yml` 替代）
3. 架构结论 §23: 模型配置、provider、env、args

建议明确三个文件的职责边界：

| 文件 | 定位 | 子系统 | 内容 |
|------|------|--------|------|
| `runtimes/manifest.yml` | 构建时真相源 | C | 版本号、SHA256、下载 URL |
| `~/.archbot/config/runtimes.yml` | 运行时配置 | B | isolation、model、provider |
| `项目/.archbot/agent.yml` | 项目级覆盖 | A | 项目级 runtime 选择、sandbox 策略 |

### 🟡 A2. 三个子系统的成熟度不一致

| 子系统 | 设计迭代次数 | 评审次数 | 成熟度 |
|--------|------------|---------|--------|
| A 编排引擎 | 3 轮 (v1 → v2 初始 → v2 最终) | 2 次 | 高 |
| B 配置隔离 | 1 轮 | 0 次 | 中 |
| C 预打包分发 | 2 轮 (初始 → OS 分版) | 0 次 | 中 |

B 和 C 的某些设计决策可能在与 A 的交互中暴露问题。建议在实现前对 B 和 C 各增加一轮独立评审。

### 🟡 A3. 33 个文件的变更原子性

统一方案涉及 19 个新文件 + 14 个修改文件。这些变更跨前端 TypeScript、后端 Rust、Shell 脚本、YML 配置、CI 配置。如果一次性提交，回滚困难。

**建议**: 按子系统分 3 个 PR：
- PR1: A 编排引擎（11 个文件）
- PR2: B 配置隔离（4 个文件）
- PR3: C 预打包分发（14 个文件）

依赖关系: PR1 独立 → PR2 独立 → PR3 依赖 PR1+PR2（分发脚本中的目录结构需与隔离方案一致）

## A4. 架构完整度评分

| 维度 | 评分 | 说明 |
