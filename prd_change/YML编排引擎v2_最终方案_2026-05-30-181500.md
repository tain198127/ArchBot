## 一、用户场景（评审 B1 已修复）

**场景 A: 安全攻击面收窄**

> 攻击者通过篡改 `actions.yml` 将 `command` 改为 `malicious_delete_all`。在 v1 中，前端直接 `invoke(command, args)` 调用，无白名单拦截。在 v2 中，ActionRegistry 的 `executeBackend()` 检查白名单 → `malicious_delete_all` 不在 Set 中 → 抛出 Security Error，操作被阻止。

**场景 B: 桌面端右键体验原生感**

> 用户在 Tauri 桌面版 ArchBot 中右键点击文件树节点。v1 弹出的是 Web DOM 右键菜单（浏览器风格，与系统割裂）。v2 检测到 Tauri 环境 → 调用 `Menu.popup()` 渲染原生右键菜单（与 VS Code / Finder 体验一致）。浏览器环境下自动降级为 Vue DOM 菜单。

**场景 C: 新增右键操作时权限自动校验**

> 开发者新增一个 `ds.exportExcel` action，在 `context-menus.yml` 中配置右键显示，但忘记在 `permissions.yml` 中注册。v1 中菜单直接显示且可点击。v2 中 ConfigValidator 启动时交叉校验 → 发现 `ds.exportExcel` 在 context-menus 中被引用但不在任何 permissions.allowActions 中 → 控制台 warning + toast 提示，菜单仍显示但 action 执行时被 ActionRegistry 拦截。

### 2.1 ActionRegistry 内置 permissions 校验（评审 B2 已修复）

**问题**: Tauri MenuItem action 回调直接调 `registry.execute()`，跳过了 permissions 层。

**修复**: 权限校验统一在 `ActionRegistry.execute()` 内部完成，不依赖调用方的 discipline：

```typescript
// src/orchestration/ActionRegistry.ts

export class ActionRegistry {
  private handlers = new Map<string, ActionHandler>()
  private allowedBackendCommands = new Set<string>()
  private permissionMap = new Map<string, Set<string>>() // resourceType → allowed action ids

  loadPermissions(permissions: PermissionRule[]): void {
    for (const rule of permissions) {
      this.permissionMap.set(rule.resourceType, new Set(rule.allowActions))
    }
  }

  async execute(
    actionId: string,
    params: Record<string, unknown>,
    runtime: ActionRuntime,
    context?: { resourceType: string },
  ): Promise<unknown> {
    // 第 2 层: 业务权限校验（v2 新增）
    if (context) {
      const allowed = this.permissionMap.get(context.resourceType)
      if (allowed && !allowed.has(actionId)) {
        runtime.pushLog('warn', 'ActionRegistry',
          `Action "${actionId}" not allowed for resourceType "${context.resourceType}"`)
        throw new Error(`[Permissions] Action not allowed: ${actionId}`)
      }
    }

    const handler = this.handlers.get(actionId)
    if (!handler) {
      throw new Error(`[ActionRegistry] Unknown action: ${actionId}`)
    }
    return handler(params, runtime)
  }

  /** 第 3 层: Backend command 白名单（v2 新增）*/
  allowBackendCommand(command: string): void {
    this.allowedBackendCommands.add(command)
  }

  async executeBackend(command: string, args: Record<string, unknown>): Promise<unknown> {
    if (!this.allowedBackendCommands.has(command)) {
      throw new Error(`[Security] Backend command not whitelisted: ${command}`)
    }
    // 实际 invoke 由 actions/*.ts 的 handler 调用 runtime.invoke()
    // 白名单只是防御层——即使 handler 被意外触发，也会拦截
    throw new Error('[ActionRegistry] executeBackend is a guard, not an executor')
  }
}
```

**关键**: permissions 校验在 `execute()` 中统一执行，无论是菜单点击、右键菜单、还是未来任何触发源，都经过同一道门。

`★ Insight ─────────────────────────────────────`
这就是"纵深防御"的精髓——不要指望调用方（MenuBar / ContextMenuPresenter / 未来的命令面板 / Agent）都记得做权限校验。把校验逻辑放在唯一的执行入口（ActionRegistry.execute），让安全成为默认行为而非约定行为。
`─────────────────────────────────────────────────`

### 2.2 ContextMenu 异步时序方案（评审 B3 已修复）

**问题**: Vue `@contextmenu.prevent` 是同步事件，Tauri `Menu.popup()` 是异步跨 JS-Rust 调用。

**方案**: `@contextmenu.prevent` 只负责阻止浏览器默认菜单。原生菜单异步弹出 — `void` 触发不等待：

```typescript
// FileTreePanel.vue 中的用法
function onContextMenu(event: MouseEvent, node: TreeNode) {
  const context = buildContextObject(node)
  // 不 await — popup 是异步的，但不需要阻塞事件处理器
  void showContextMenu(context, event.clientX, event.clientY)
}

// ContextMenuPresenter.ts
export async function showContextMenu(
  context: ContextObject,
  x: number,
  y: number,
): Promise<void> {
  const items = getContextMenuItems(context)
  const filtered = filterByPredicates(items, getRuntimeState(), context)

  if (isTauri()) {
    try {
      await showNativeContextMenu(filtered, context, x, y)
    } catch (err) {
      // 原生菜单创建失败 → 降级 Vue DOM
      pushLog('warn', 'ContextMenu', `Native menu failed, fallback to DOM: ${err}`)
      showVueContextMenu(filtered, context, x, y)
    }
  } else {
    showVueContextMenu(filtered, context, x, y)
  }
}
```

**为什么不用 await**: `@contextmenu.prevent` 处理后立即返回，浏览器不再等待。Tauri 的原生菜单在 Rust 侧异步创建并弹出，用户看到的菜单是操作系统级别的 UI 元素，不依赖 JS 事件循环的持续运行。

### 2.3 NativeMenuFactory 依赖注入（评审 B4 已修复）

**问题**: `@tauri-apps/api/menu` 在 vitest 环境不可用，需要 mock 接口。

**方案**:

```typescript
// src/orchestration/NativeMenuFactory.ts

export interface NativeMenuItem {
  id?: string
  text: string
  enabled: boolean
  action?: () => Promise<void>
  isSeparator?: boolean
}

export interface NativeMenu {
  popup(x: number, y: number): Promise<void>
}

export interface NativeMenuFactory {
  createMenu(items: NativeMenuItem[]): Promise<NativeMenu>
}

// 生产实现（Tauri 桌面环境）
let factory: NativeMenuFactory = {
  async createMenu(items: NativeMenuItem[]): Promise<NativeMenu> {
    const { Menu, MenuItem, PredefinedMenuItem } = await import('@tauri-apps/api/menu')
    const menuItems = []
    for (const item of items) {
      if (item.isSeparator) {
        menuItems.push(await PredefinedMenuItem.new({ item: 'Separator' }))
      } else {
        menuItems.push(await MenuItem.new({
          id: item.id,
          text: item.text,
          enabled: item.enabled,
          action: item.action,
        }))
      }
    }
    const menu = await Menu.new({ items: menuItems })
    return {
      async popup(x: number, y: number) {
        await menu.popup({ x, y })
      },
    }
  },
}

// 仅测试用
export function overrideNativeMenuFactory(f: NativeMenuFactory): void {
  factory = f
}

export function getNativeMenuFactory(): NativeMenuFactory {
  return factory
}
```

测试中注入 mock:

### 2.4 交叉校验 3 条规则（评审 S1）

`ConfigValidator` 新增 `crossValidate()` 方法，启动时执行：

```
规则 1 (ref-integrity): context-menus.yml / menus.yml 中引用的每个 action id，
                       必须存在于 actions.yml 中。
规则 2 (perm-coverage): actions.yml 中的每个 action id，如果被 context-menus.yml
                        引用，则必须出现在 permissions.yml 的至少一个 allowActions 中。
规则 3 (perm-ref): permissions.yml 中引用的每个 action id，必须存在于 actions.yml 中。

违反规则 1 → Error (拒绝启动)
违反规则 2 → Warning (不阻塞，但 toast 提示 + 控制台输出)
违反规则 3 → Error (拒绝启动)
```

实现位置: `src/orchestration/ConfigValidator.ts` 新增 `crossValidate(menus, contextMenus, actions, permissions): ValidationResult[]`。

### 2.5 predicates.yml 的 RuntimeState 注入（评审 S2）

predicates.yml 中 `expression` 型条件的求值路径：

```
右键点击 → ContextMenuResolver.getContextMenuItems(context)
  → 返回所有匹配 resourceType + semanticType 的菜单项
  → ContextMenuPresenter 对每个 item 调用 evaluatePredicate(item, state, context)
    → 检查 item.predicate 是否存在
    → 从 PredicateRegistry 查找 pred.id
    → 如果是 expression 型: 调用 ExpressionEvaluator.evaluateExpression(pred.expression, state, context)
    → 如果是 handler 型: 调用 PredicateRegistry.evaluate(pred.id, state, context)
```

**关键设计决策**: RuntimeState 由调用方（FileTreePanel 等）传入，而非在 ContextMenuResolver 内部获取。这样 resolver 保持纯函数特性，测试时 mock state 更容易。

```typescript
export function filterByPredicates(
  items: ContextMenuItem[],
  state: RuntimeState,
  context: ContextObject,
): ContextMenuItem[] {
  const predicates = getConfig().predicates // 从 ConfigLoader 获取
  return items.filter(item => {
    if (!item.predicate) return true

    const pred = predicates?.find(p => p.id === item.predicate)
    if (!pred) return false // 未注册的 predicate → 安全侧：不显示

    if (pred.expression) {
      return evaluateExpression(pred.expression, state, context)
    }
    if (pred.handler) {
      return getPredicateRegistry().evaluate(pred.handler, state, context)
    }
    return false
  })
}
```

### 2.6 Provider 动态菜单的 label 策略（评审 S3）

Provider 返回的菜单项 label 分两类：

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| i18n key（固定文本） | `"menuFile.recentProjects"` | 渲染时调用 `tt(label)` |
| 运行时数据（动态名称） | `"my-spring-project"` | 原样展示，不经过 `tt()` |

**区分规则**: label 以 `menu` / `contextMenu` / `common` 开头 → i18n key，否则为运行时数据。在 MenuBar / ContextMenuPresenter 渲染时判断：

```typescript
function resolveLabel(label: string): string {
  return /^(menu|contextMenu|common|license)\./.test(label) ? tt(label) : label
}
```

## 五、安全模型（五层，评审确认）

```
Layer 1: visibleWhen/enabledWhen          = UX 控制       (YML 条件表达式)
Layer 2: permissions.yml allowActions     = 业务权限       (YML 配置 → ActionRegistry.execute 前置校验)
Layer 3: ActionRegistry allowedCommands   = 入口白名单     (TS Set → 启动时注册)
Layer 4: Tauri Capability + Permission    = 系统沙箱       (JSON capability 文件)
Layer 5: Rust Command 参数校验             = 资源守卫       (Rust 函数内校验)

每一层独立校验，不依赖外层。
即使 Layer 1 菜单显示，Layer 2 仍会拦截。
即使 Layer 2 被绕过，Layer 3 仍会拦截未注册的 backend command。
```
