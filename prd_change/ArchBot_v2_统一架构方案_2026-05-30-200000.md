## 1.2 用户场景串联

**场景: 用户在 macOS 上安装 ArchBot，右键 Controller 文件进行完整调用链分析**

```
1. 用户下载 ArchBot-2.0.0-darwin-arm64.dmg (~310MB)
   └── 内含 4 个 Runtime + 4 个 Adapter .tar.gz          ← C. 预打包分发

2. 首次启动，ArchBot 解压 Runtime 到 ~/.archbot/runtimes/
   └── 初始化隔离 HOME + config 目录                      ← B. 配置隔离

3. 用户打开项目，右键 Controller 文件
   └── ContextMenuResolver 匹配 context-menus.yml         ← A. 编排引擎
   └── Menu.popup() 弹出原生右键菜单（Tauri）              ← A. 编排引擎

4. 用户点击"完整分析调用链"
   └── ActionRegistry.execute() 校验 permissions          ← A. 编排引擎 (Layer 2)
   └── FlowExecutor 串行执行 7 个步骤                      ← A. 编排引擎
   └── 第 4 步调用 backend command → 白名单校验             ← A. 编排引擎 (Layer 3)
   └── Runtime 进程以隔离 HOME 启动                        ← B. 配置隔离
   └── 文件访问审计 running...                             ← B. 配置隔离 (Layer 3)

5. Turn 结束
   └── 审计扫描 → 无违规访问                               ← B. 配置隔离
   └── 结果刷新到调用链面板
```

## A.2 v2 新增能力

### A.2.1 ActionRegistry 内置 permissions 校验

权限校验统一在 `ActionRegistry.execute()` 内部完成：

```typescript
async execute(actionId, params, runtime, context?: { resourceType: string }) {
  // Layer 2: 业务权限校验
  if (context) {
    const allowed = this.permissionMap.get(context.resourceType)
    if (allowed && !allowed.has(actionId)) {
      throw new Error(`[Permissions] Action not allowed: ${actionId}`)
    }
  }
  // Layer 3: Backend command 白名单（由 actions/*.ts 注册时声明）
  const handler = this.handlers.get(actionId)
  if (!handler) throw new Error(`[ActionRegistry] Unknown action: ${actionId}`)
  return handler(params, runtime)
}
```

### A.2.3 ContextMenuPresenter（Tauri 原生 + Vue 降级）

```typescript
export async function showContextMenu(context, x, y) {
  const items = getContextMenuItems(context)
  const filtered = filterByPredicates(items, getRuntimeState(), context)

  if (isTauri()) {
    try { await showNativeContextMenu(filtered, context, x, y) }
    catch { showVueContextMenu(filtered, context, x, y) }  // 降级
  } else {
    showVueContextMenu(filtered, context, x, y)
  }
}
```

### A.2.4 交叉校验

| 规则 | 违反后果 |

# 第三部分: B. 配置隔离（Runtime Sandbox）

## B.1 三层强制执行模型

```
第 1 层: 环境变量重定向     CLAUDE_CONFIG_DIR, HERMES_HOME, ...
         告知 Runtime "正确" 的配置路径。弱点：无法阻止硬编码路径的代码。

第 2 层: 进程级沙箱         HOME 重写 + env_clear + 白名单注入 + macOS Sandbox
         让 Runtime 进程活在隔离的文件系统命名空间中。            ← 核心新增

第 3 层: 文件访问审计       Turn 结束后扫描 Runtime 访问的文件路径
         发现前两层未拦截的违规访问。
```

## B.2 Runtime Launcher（Rust 实现）

```rust
pub fn launch_isolated_runtime(config: RuntimeLaunchConfig) -> Result<Child, String> {
    let mut cmd = Command::new(&config.executable);

    cmd.env_clear();                                        // 1. 清空继承环境
    cmd.env("HOME", &config.isolated_home);                 // 2. 注入隔离 HOME
    for (k, v) in &config.allowed_env { cmd.env(k, v); }    // 3. 白名单环境变量
    cmd.env("PATH", "/usr/local/bin:/usr/bin:/bin");        // 4. 最小系统环境
    cmd.current_dir(&config.workspace_root);                // 5. 工作目录

    cmd.spawn()
}
```

关键效果：Runtime 进程的 `$HOME` = `~/.archbot/runtimes/claude-code/home/`。即使内部硬编码了 `~/.claude`，读到的也是隔离目录下的空文件夹。

## B.3 隔离目录结构

```
~/.archbot/runtimes/
├── claude-code/
│   ├── versions/2.1.128/     ← 从 .tar.gz 解压
│   ├── current -> versions/2.1.128
│   ├── config/               ← CLAUDE_CONFIG_DIR
│   ├── home/                 ← 隔离 HOME ($HOME)
│   └── logs/

```

## C.3 关键约束

- **不能交叉编译 macOS ARM64 ↔ x64**: Hermes 的 Python C 扩展依赖原生 wheel，必须各用物理机构建
- **manifest.yml 是唯一真相源**: 版本号、SHA256、下载源全部锁定在此文件中
- **不在用户机器上执行 npm/pip/brew install**: 所有 Runtime 都是预构建的 `.tar.gz`

# 第五部分: 统一安全模型

```
Layer 1: visibleWhen/enabledWhen          = UX 控制       (YML — A 子系统)
Layer 2: permissions.yml allowActions     = 业务权限       (YML → ActionRegistry — A 子系统)
Layer 3: ActionRegistry allowedCommands   = 入口白名单     (TS Set — A 子系统)
Layer 4: HOME 重写 + Tauri Capability     = 进程沙箱       (Rust launcher — B 子系统)
Layer 5: Rust Command 参数校验             = 资源守卫       (Rust — B 子系统)
```

三层互补，缺一不可：
- A 子系统控制 "谁能调什么动作"
- B 子系统控制 "进程能看到什么文件"
- C 子系统保证 "包在分发链上未被篡改"
