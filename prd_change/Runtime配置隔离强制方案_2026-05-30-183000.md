## 一、问题陈述

架构结论 §17-23 声明了各 Runtime 应使用 ArchBot 托管的隔离配置目录，§32 列出了禁止访问的路径。但这些是**声明式策略**——规定了目标，没有规定强制执行机制。

核心问题：

> 单靠 `CLAUDE_CONFIG_DIR=...` 环境变量是一种**约定**，不是**强制**。如果 Runtime 代码中存在硬编码的 `~/.claude` 路径，或者 Runtime 的某个依赖库、插件、子进程无视了环境变量，如何保证它不会读取或污染用户全局配置？

### 场景

| 场景 | 风险 |
|------|------|
| Claude Code 内部某模块硬编码了 `~/.claude/settings.json` | 读了用户全局配置，可能泄露 API key 或自定义规则 |
| Hermes 的 Python 依赖读了 `~/.hermes/memory/` | 不同项目间的会话记忆交叉污染 |
| OpenClaw 的插件写了 `~/.openclaw/plugins/cache/` | 污染用户全局 OpenClaw 配置 |
| Runtime 的子进程（如 git、npm）读了 `~/.gitconfig` | 暴露用户身份信息（姓名、邮箱） |

### 成功度量

| 度量 | 目标 |
|------|------|
| 每个 Runtime 进程看到的 `HOME` | `~/.archbot/runtimes/<runtime>/home/`，非用户真实 HOME |
| `~/.claude` 访问拦截率 | 100%（Runtime 进程不应感知到真实 `~/.claude` 的存在） |
| 隔离违规检测延迟 | ≤ 1 turn（Turn 结束后审计扫描发现） |
| 跨项目配置泄漏 | 0 次 |

## 二、三层强制执行模型

```
┌──────────────────────────────────────────────────────────────┐
│ 第 1 层: 环境变量重定向（配置层面）                           │
│   CLAUDE_CONFIG_DIR, HERMES_HOME, ...                        │
│   职责: 告知 Runtime "正确" 的配置路径                         │
│   弱点: 无法阻止硬编码路径的代码                               │
├──────────────────────────────────────────────────────────────┤
│ 第 2 层: 进程级沙箱（系统层面）          ← 本次新增核心        │
│   HOME 重写 + 环境变量白名单 + PATH 隔离 + macOS Sandbox      │
│   职责: 让 Runtime 进程活在隔离的文件系统命名空间中             │
│   弱点: 内核级沙箱依赖平台能力（macOS Sandbox / Linux bwrap）   │
├──────────────────────────────────────────────────────────────┤
│ 第 3 层: 文件访问审计（检测层面）          ← 本次新增           │
│   Turn 结束后扫描 Runtime 访问的文件路径                       │
│   职责: 发现前两层未拦截的违规访问                             │
│   弱点: 事后检测，不能实时阻止                                 │
└──────────────────────────────────────────────────────────────┘
```

三层互补：第 1 层是"请走正门"，第 2 层是"封死侧门"，第 3 层是"巡逻发现翻墙"。

## 四、第 2 层：进程级沙箱（核心新增）

### 4.1 总体原则

```
Runtime 进程不应感知用户真实 HOME。
Runtime 进程不应继承用户 shell 环境。
Runtime 进程只能看到 ArchBot 显式注入的目录。
```

### 4.2 Runtime Launcher（Rust 实现）

```rust
// src-tauri/src/agent_runtime/launcher.rs

use std::collections::HashMap;
use std::process::{Child, Command};

pub struct RuntimeLaunchConfig {
    /// Runtime 类型标识
    pub runtime_type: String,
    /// 可执行文件路径（ArchBot 托管版本）
    pub executable: String,
    /// 工作目录（项目根目录）
    pub workspace_root: String,
    /// ArchBot 管理的隔离 HOME
    pub isolated_home: String,
    /// 显式允许的环境变量（白名单）
    pub allowed_env: HashMap<String, String>,
    /// 启动参数
    pub args: Vec<String>,
    /// 超时秒数
    pub timeout_seconds: u64,
}

/// 启动一个被隔离的 Runtime 进程
pub fn launch_isolated_runtime(config: RuntimeLaunchConfig) -> Result<Child, String> {
    let mut cmd = Command::new(&config.executable);

    // ═══════════════════════════════════════════════
    // 1. 清空所有继承的环境变量
    // ═══════════════════════════════════════════════
    cmd.env_clear();

    // ═══════════════════════════════════════════════
    // 2. 注入隔离 HOME（最关键的一步）
    //    Runtime 内所有 ~/ 路径解析到这里
    //    ~/.claude → ~/.archbot/runtimes/claude-code/home/.claude
    //    ~/.hermes → ~/.archbot/runtimes/hermes/home/.hermes
    // ═══════════════════════════════════════════════
    cmd.env("HOME", &config.isolated_home);

    // ═══════════════════════════════════════════════
    // 3. 只注入白名单环境变量
    // ═══════════════════════════════════════════════
    for (key, value) in &config.allowed_env {
        cmd.env(key, value);
    }

    // ═══════════════════════════════════════════════
    // 4. 最小系统环境（不含用户 shell profile）
    // ═══════════════════════════════════════════════
    cmd.env("PATH", "/usr/local/bin:/usr/bin:/bin");  // 不含用户自定义 PATH
    cmd.env("LANG", "en_US.UTF-8");
    cmd.env("TZ", "UTC");

    // 不注入: USER, LOGNAME, SHELL, TMPDIR（用户相关）
    // 不注入: SSH_AUTH_SOCK, GPG_AGENT_INFO（安全凭据）
    // 不注入: NPM_CONFIG_USERCONFIG, PIP_CONFIG_FILE（除非白名单声明）

    // ═══════════════════════════════════════════════
    // 5. 设置工作目录
    // ═══════════════════════════════════════════════
    cmd.current_dir(&config.workspace_root);

    // ═══════════════════════════════════════════════
    // 6. 传递启动参数
    // ═══════════════════════════════════════════════
    cmd.args(&config.args);

    // ═══════════════════════════════════════════════
    // 7. 平台特定沙箱（可选增强）
    // ═══════════════════════════════════════════════
    #[cfg(target_os = "macos")]
    {
        // macOS: 如果 ArchBot 自身启用了 App Sandbox，
        // 子进程自动继承沙箱限制。
        // 无需额外配置，但需要 ArchBot 的 entitlements 中声明
        // com.apple.security.app-sandbox = true
    }

    #[cfg(target_os = "linux")]
    {
        // Linux: 可选使用 bubblewrap 做更严格的隔离
        // bwrap --bind ~/.archbot/runtimes/claude-code/home /home/sandbox
        //       --bind /project/dir /project/dir
        //       --tmpfs /tmp
        //       --unshare-all
        //       claude ...
        // 第一版不做 bwrap，依赖 HOME 重写 + 审计。
    }

    cmd.spawn().map_err(|e| format!("Failed to launch {}: {}", config.runtime_type, e))
}
```

`★ Insight ─────────────────────────────────────`
`cmd.env_clear()` 是关键的第一步。如果不先清空，Runtime 进程会继承 ArchBot 进程的全部环境变量——包括 `HOME`、`USER`、`SSH_AUTH_SOCK`、各种 API key。清空后只注入白名单，确保 Runtime 看不到它不该看到的东西。这和 Docker 的 `--env` 白名单模式是同一个思想：默认拒绝，显式允许。
`─────────────────────────────────────────────────`

### 4.4 隔离 HOME 的必要性论证

即使设置了 `CLAUDE_CONFIG_DIR=...`，以下场景仍会触及 `~/.claude`：

| 场景 | 原因 | 隔离 HOME 的效果 |
|------|------|-----------------|
| Claude Code CLI 内部有 fallback 逻辑 | 如果 `CLAUDE_CONFIG_DIR` 不存在，可能回退到 `~/.claude` | 隔离 HOME 下不存在 `~/.claude`，Runtime 会创建一个新的空目录而非读取用户真实配置 |
| Claude Code 的 npm 全局包读到用户配置 | npm 可能读 `~/.npmrc` | 隔离 HOME 下无 `.npmrc` |
| git 操作读到用户身份 | git 读 `~/.gitconfig` 获取 user.name/user.email | 隔离 HOME 下无 `.gitconfig` → git 报错而非泄露身份 |
| Python 包读到用户 pip 配置 | pip 读 `~/.pip/pip.conf` 或 `~/.config/pip/` | 隔离 HOME 下无这些目录 |

### 4.5 需要桥接的合法访问

有些 `~/` 下的文件是 Runtime 正常工作必需的，需要显式桥接：

```rust
// launcher.rs 中——在清空环境后，显式挂载需要的文件

        let git_name = config.project_git_user.clone().unwrap_or("ArchBot".into());
```

## 五、第 3 层：文件访问审计

### 5.1 审计时机

每次 Turn 结束后，ArchBot 的 `AgentAuditManager` 扫描该 Turn 期间 Runtime 进程访问过的文件。

### 5.2 审计实现

```rust
// src-tauri/src/agent_runtime/audit.rs

use std::path::Path;

pub struct AuditRule {
    /// 禁止访问的路径前缀
    pub forbidden_prefix: String,
    /// 严重级别
    pub severity: AuditSeverity,
}

pub enum AuditSeverity {
    Critical,  // 读到用户凭据（~/.ssh, ~/.aws）
    High,      // 读到其他 Runtime 的配置（~/.claude, ~/.hermes）
    Medium,    // 读到用户个人数据（~/.config, ~/.gitconfig）
    Low,       // 读到无关系统文件
}

    /// 默认审计规则（对齐架构结论 §32）
    pub fn default_rules() -> Vec<AuditRule> {
        vec![

            AuditRule::new("~/.aws",            AuditSeverity::Critical),
            AuditRule::new("~/.claude",         AuditSeverity::High),
            AuditRule::new("~/.hermes",         AuditSeverity::High),
            AuditRule::new("~/.opencode",       AuditSeverity::High),
```

### 5.3 审计响应

| 严重级别 | 响应 |
|---------|------|
| Critical | 立即终止 Turn + 记录审计日志 + toast 告警用户 |
| High | 完成当前 Turn + 记录审计日志 + 下次 Turn 前 toast 告警 |
| Medium | 记录审计日志，不中断 |
| Low | 仅记录日志 |

### 5.4 第一版实现策略

第三层的完整实现依赖 OS 级文件监控（macOS Endpoint Security / Linux fanotify），第一版可以简化：

1. **Dtrace/Strace 包装**: 启动 Runtime 时通过 `dtruss`（macOS）或 `strace`（Linux）捕获文件相关的 syscall
2. **/proc 扫描**: Linux 上定期扫描 `/proc/<pid>/fd/` 中的符号链接目标
3. **事后扫描**: Turn 结束后 `find` 用户真实 HOME 下的关键目录（`~/.claude`、`~/.ssh`），检查是否有新的修改时间戳

## 七、setup_isolated_home() 初始化逻辑

在 Runtime 首次启动前，ArchBot 需要为它准备隔离 HOME：

```rust
// src-tauri/src/agent_runtime/home_setup.rs

use std::fs;
use std::path::PathBuf;

pub struct IsolatedHomeConfig {
    pub home_path: PathBuf,
    pub needs_git: bool,
    pub git_user_name: Option<String>,
    pub git_user_email: Option<String>,
    pub needs_ssh: bool,
    pub ssh_key_path: Option<PathBuf>,  // from ArchBot Secret Manager
}

pub fn setup_isolated_home(config: &IsolatedHomeConfig) -> Result<(), String> {
    // 1. 创建隔离 HOME 目录
    fs::create_dir_all(&config.home_path)
        .map_err(|e| format!("Failed to create isolated HOME: {}", e))?;

    // 2. 如果需要 git，生成 .gitconfig
    if config.needs_git {
        let gitconfig = format!(
            "[user]\n  name = {}\n  email = {}\n",
            config.git_user_name.as_deref().unwrap_or("ArchBot"),
            config.git_user_email.as_deref().unwrap_or("archbot@local"),
        );
        fs::write(config.home_path.join(".gitconfig"), gitconfig)
            .map_err(|e| format!("Failed to write .gitconfig: {}", e))?;
    }

    // 3. 如果需要 SSH，从 Secret Manager 获取 deploy key
    if config.needs_ssh {
        if let Some(key_path) = &config.ssh_key_path {
            let ssh_dir = config.home_path.join(".ssh");
            fs::create_dir_all(&ssh_dir)
                .map_err(|e| format!("Failed to create .ssh: {}", e))?;
            fs::copy(key_path, ssh_dir.join("id_ed25519"))
                .map_err(|e| format!("Failed to copy SSH key: {}", e))?;
            // 设置权限 600
            #[cfg(unix)]
            {
                use std::os::unix::fs::PermissionsExt;
                fs::set_permissions(ssh_dir.join("id_ed25519"),
                    fs::Permissions::from_mode(0o600))
                    .map_err(|e| format!("Failed to set SSH key permissions: {}", e))?;
            }
        }
    }

    Ok(())
}
```

## 八、测试策略

### 8.1 单元测试

### 8.2 集成测试

| 测试 | 验证点 |
|------|--------|
| 启动 Claude Code Runtime → 检查实际 HOME | 在 Runtime 内执行 `echo $HOME`，验证不是用户真实 HOME |
| Claude Code 读取配置 | 检查它访问的 settings.json 路径在隔离目录下 |
| Claude Code 写入 memory | 检查写入的 memory 文件在 `~/.archbot/runtimes/claude-code/current/config/memory/` 而非 `~/.claude/memory/` |
| Hermes pip install | 检查包安装在 venv 下，非系统 Python |
| 违规访问审计 | 模拟 Runtime 进程读 `~/.claude` → 审计日志应捕获 |

### runtimes.yml 新增配置段

在架构结论 §23 的 `runtimes.yml` 中，每个 Runtime 的 `execution` 段增加 `isolation` 配置：

```yaml
# 以 claude_code 为例
claude_code:
  # ... 已有配置 ...

  execution:
    default_timeout_seconds: 1800
    max_concurrent_turns: 2
    working_dir_policy: project_root
    inherit_system_env: false    # 已有 → 映射到 cmd.env_clear()

    isolation:                   # ← 新增
      mode: strict               # strict | permissive
      isolated_home: "~/.archbot/runtimes/claude-code/home"
      env_policy: whitelist      # whitelist | inherit_filtered
      allowed_env_keys:          # 显式白名单
        - CLAUDE_CONFIG_DIR
        - ANTHROPIC_BASE_URL
        - ANTHROPIC_AUTH_TOKEN
        - ANTHROPIC_MODEL
        - PATH
        - LANG
      bridge:
        git: true                # 在隔离 HOME 生成 .gitconfig
        ssh: false               # 不桥接 SSH key
      audit:
        enabled: true
        severity_threshold: medium  # critical | high | medium | low
```

## 十、不做

- 完整的 Linux `bwrap`/`firejail` 集成（第一版依赖 HOME 重写 + 审计，后续版本可选）

- 网络隔离（Runtime 需要访问模型 API）
