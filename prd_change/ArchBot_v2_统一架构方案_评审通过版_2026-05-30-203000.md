## A.1 v2 新增能力

### ActionRegistry 内置 permissions 校验

```typescript
async execute(actionId, params, runtime, context?: { resourceType: string }) {
  // Layer 2: 业务权限
  if (context) {
    const allowed = this.permissionMap.get(context.resourceType)
    if (allowed && !allowed.has(actionId))
      throw new Error(`[Permissions] Action not allowed: ${actionId}`)
  }
  // Layer 3: 白名单 + handler lookup + 执行
}
```

## B.2 Runtime Launcher（评审 B2 已修复）

```rust
pub fn launch_isolated_runtime(config: RuntimeLaunchConfig) -> Result<Child, String> {
    let mut cmd = Command::new(&config.executable);

    // 1. 清空继承环境
    cmd.env_clear();

    // 2. 注入隔离 HOME
    cmd.env("HOME", &config.isolated_home);

    // 3. 只注入白名单
    for (k, v) in &config.allowed_env { cmd.env(k, v); }

    // 4. 最小系统环境
    cmd.env("PATH", "/usr/local/bin:/usr/bin:/bin");
    cmd.env("LANG", "en_US.UTF-8");
    cmd.env("TZ", "UTC");

    // 5. Windows 关键系统变量（评审 B2 修复）
    #[cfg(target_os = "windows")]
    {
        cmd.env("SystemRoot", "C:\\Windows");
        cmd.env("TEMP", &config.isolated_temp);
        cmd.env("TMP", &config.isolated_temp);
        // 不注入 USERPROFILE — 用隔离 HOME 替代
    }

    cmd.current_dir(&config.workspace_root);
    cmd.args(&config.args);
    cmd.spawn()
}
```

# 第五部分: Adapter 协议契约（评审 B3 已修复）

> 架构结论 §7-8 定义了 Adapter HTTP API。本节提取 A/B/C 三个子系统各自依赖的接口子集，确保实现时接口假设一致。

## 5.1 接口总览

```
Adapter 暴露给 ArchBot 的 HTTP API:

  GET  /v1/runtime/health              → { status, version }
  GET  /v1/runtime/capabilities        → { capabilities[] }
  POST /v1/sessions                    → { session_id }
  GET  /v1/sessions/{id}               → { session }
  POST /v1/sessions/{id}/turns         → { turn_id }        ← 核心
  GET  /v1/sessions/{id}/turns/{tid}   → { turn }
  POST /v1/sessions/{id}/turns/{tid}/cancel
  GET  /v1/sessions/{id}/turns/{tid}/events  (SSE)
```

## 5.2 子系统依赖矩阵

| 接口 | A 编排引擎 | B 配置隔离 | C 预打包 |
|------|-----------|-----------|---------|
| `/v1/runtime/health` | — | ✅ 启动前健康检查 | ✅ 构建后可用性测试 |
| `/v1/runtime/capabilities` | ✅ 注册 action 时查询能力 | — | — |
| `POST /v1/sessions` | ✅ 用户触发 Turn | ✅ 注入隔离配置 | — |
| `POST .../turns` | ✅ Flow 执行入口 | ✅ 准备隔离 HOME | — |
| `GET .../events` (SSE) | ✅ 流式展示 | ✅ 审计事件流 | — |
| `POST .../cancel` | ✅ 用户取消 | — | — |

## 5.3 Adapter 启动契约

```
ArchBot 启动 Adapter 时注入:
  adapter_id, runtime_type, runtime_version,
  listen_host (127.0.0.1), listen_port (auto),
  auth_token (startup_token),
  config_path, workspace_root, log_dir

Adapter 就绪后向 stdout 输出:
  {"status":"ready","host":"127.0.0.1","port":18790}

ArchBot 捕获 stdout 行 → 记录端口 → 后续 HTTP 调用使用此端口
```

## 5.4 Adapter 与 Runtime 的关系

```
ArchBot ←→ HTTP+SSE ←→ Adapter ←→ SDK/API/CLI ←→ Runtime

Adapter 负责:
  1. 封装 Runtime SDK/API/CLI
  2. 把 Runtime 事件转换为 ArchBot 标准事件
  3. 把 Runtime 错误转换为 ArchBot 标准错误
  4. 屏蔽 Runtime 版本差异

ArchBot 不直接 import Runtime SDK。
```

## 5.5 Runtime 调用方式（逐 Runtime 具体命令）

> 架构结论 §39 定义了优先级：**SDK 优先 → API 次之 → CLI fallback 最后**。
> 以下是在第一版中实际使用的调用方式。

### 5.5.1 Claude Code

**接入方式**: CLI（Claude Code 目前不提供 SDK，native binary 是最稳定的接口）

```bash
# Adapter 内部执行的命令
~/.archbot/runtimes/claude-code/current/claude \
  --input-file .archbot/agent/sessions/sess_001/turns/turn_008/input.yml \
  --prompt-file .archbot/agent/sessions/sess_001/turns/turn_008/prompt.txt \
  --output-dir .archbot/agent/sessions/sess_001/turns/turn_008/ \
  --model "${ANTHROPIC_MODEL}" \
  --max-turns 1 \
  --no-interactive \
  --json-output
  # --json-output: 输出 NDJSON，每行一个事件，方便 Adapter 解析
```

| 参数 | 含义 |
|------|------|
| `--input-file` | 结构化输入（input.yml 路径） |
| `--prompt-file` | 启动指令（prompt.txt 路径） |
| `--output-dir` | 结果文件输出目录 |
| `--model` | 模型名称（由 ArchBot runtimes.yml 注入） |
| `--max-turns 1` | 单轮执行，不支持 Agent 自主多轮循环 |
| `--no-interactive` | 非交互模式，执行完即退出 |
| `--json-output` | NDJSON 事件流输出到 stdout |

**输出捕获**:

```
Claude Code stdout (NDJSON):
{"type":"turn.delta","text":"分析 UserController..."}
{"type":"turn.tool_started","tool":"Read","path":"UserController.java"}
{"type":"turn.tool_finished","tool":"Read"}
{"type":"turn.completed"}
{"type":"turn.file_changed","path":"result.md"}
```

Adapter 将 NDJSON 行转换为 ArchBot SSE 事件：
```
NDJSON → Adapter 解析 → ArchBot SSE event
{"type":"turn.delta",...}     → event: turn.delta
{"type":"turn.completed",...} → event: turn.completed
```

**结构化结果文件**（Claude Code 写入 `--output-dir`）:
```
.archbot/agent/sessions/sess_001/turns/turn_008/
  ├── input.yml
  ├── prompt.txt
  ├── stdout.log          ← Claude Code 原始 stdout（NDJSON）
  ├── stderr.log          ← Claude Code 原始 stderr
  ├── result.md           ← Markdown 格式的最终结果
  ├── result.json         ← 结构化最终结果
  ├── file_changes.json   ← 文件变更记录
  └── events.jsonl        ← Adapter 转换后的 ArchBot 标准事件
```

### 5.5.5 Adapter 标准化流程（四个 Runtime 通用）

```
┌──────────────────────────────────────────────────────────────┐
│                    Adapter 内部流程                           │
│                                                              │
│  1. 接收 POST /v1/sessions/{id}/turns                        │
│     └── 获取 input_file, prompt_file, workspace_root         │
│                                                              │
│  2. 构造 Runtime CLI 命令（按 5.5.1-5.5.4 的格式）           │
│     └── 注入隔离 HOME 环境变量（由 B 子系统提供）              │
│                                                              │
│  3. spawn Runtime 子进程                                     │
│     └── 捕获 stdout（NDJSON/JSON lines）                      │
│     └── 捕获 stderr（错误日志）                                │
│                                                              │
│  4. 逐行解析 stdout                                          │
│     └── Runtime 原生事件 → ArchBot 标准事件                   │
│     └── 通过 SSE 推送给 ArchBot (GET .../events)              │
│                                                              │
│  5. Runtime 进程退出后                                        │
│     └── 扫描 output-dir 中的 result.md / result.json          │
│     └── 扫描 file_changes.json（文件变更）                     │
│     └── 写入 events.jsonl（存档用）                            │
│     └── 返回 Turn 状态给 ArchBot                              │
└──────────────────────────────────────────────────────────────┘
```

| 首次安装时间 | `~/.archbot/runtimes/<runtime>/metadata.json` |

## 6.2 多项目切换（评审 S2 已修复）

```
用户关闭项目 A:
  → ActionRegistry.execute('project.close')
  → 若 Runtime 进程有活跃 Turn → 等待完成或 cancel
  → Runtime 进程保持运行（复用，不重启）
  → 隔离 HOME 不变
  → 审计日志标记 project_id 切换

用户打开项目 B:
  → 切换 workspace_root
  → Runtime 进程继续使用同一隔离 HOME
  → 新一轮 Turn 在项目 B 的工作目录执行
```

Runtime 进程生命周期：**跟随 ArchBot 进程**，不跟随项目打开/关闭。
