## 三、各交付项详细设计

### 3.1 Backend Command 白名单（🔴 安全加固）

**问题**: 当前 actions.yml 中 `executor: backend` + `command: xxx` 的配置，前端直接通过 `invoke(command, args)` 调用。如果 YML 被篡改（或配置错误），可能调用到未预期的 Rust command。

**设计**:

```typescript
// src/orchestration/ActionRegistry.ts 增加白名单

class ActionRegistry {
  private handlers = new Map<string, ActionHandler>()
  private allowedBackendCommands = new Set<string>([...])

  allowBackendCommand(command: string): void {
    this.allowedBackendCommands.add(command)
  }

  // 在执行 backend action 前校验
  async executeBackend(command: string, args: Record<string, unknown>): Promise<unknown> {
    if (!this.allowedBackendCommands.has(command)) {
      throw new Error(`[Security] Backend command not whitelisted: ${command}`)
    }
    return this.runtime.invoke(command, args)
  }
}
```

**注册时机**: 应用启动时，由各 `actions/*.ts` 模块声明自己需要的 backend command：

```typescript
// src/actions/projectActions.ts
export function registerProjectActions(registry: ActionRegistry) {
  registry.allowBackendCommand('open_project')
  registry.allowBackendCommand('create_project')
  registry.register('project.open', async (params, runtime) => {
    return registry.executeBackend('open_project', { path: params.path })
  })
}
```

**白名单来源**: 由各 actions 模块在注册时显式声明 + 启动时从 actions.yml 提取所有 `executor: backend` 的 command → 交叉验证。

### 3.2 `predicates.yml`（🆕 新配置）

**问题**: 复杂条件目前需要写入 TS 代码（PredicateRegistry.register），散落在各 actions 文件中，不易发现和管理。

**设计**:

```yaml
# src/config/predicates.yml
version: "1.0"

predicates:
  - id: canAnalyzeSelectedController
    description: 项目已加载 且 右键对象为 Java Controller 文件
    expression: project.loaded == true && context.semanticType == "controller"

  - id: hasAnalysisResult
    description: 存在接口分析结果
    expression: analysis.interfaceResult.exists == true

  - id: projectIsJava
    description: 当前项目语言为 Java
    expression: project.language == "java"

  - id: groupSpecificAction
    description: 分组节点的上下文特定动作（由代码注册）
    handler: groupSpecificAction
```

**两种模式**:
- `expression`: 简单表达式（L1 语法），由 ExpressionEvaluator 求值
- `handler`: 复杂条件，通过 PredicateRegistry 注册代码实现

**加载**: ConfigLoader 加载 `predicates.yml` → `expression` 类型的存入 ExpressionEvaluator 缓存，`handler` 类型的在 PredicateRegistry 中查找对应实现 → 缺失时启动报错。

### 3.3 `permissions.yml`（🆕 新配置）

**问题**: PRD 明确要求"不要让 YML 任意指定 command"，当前只通过菜单显隐控制 UX，缺少"即使配置了 action，也不允许执行"的权限层。

**设计**:

```yaml
# src/config/permissions.yml
version: "1.0"

permissions:
  # 每种资源类型 → 允许的动作列表
  - resourceType: file
    allowActions:
      - editor.openFile
      - fs.deleteFile

  - resourceType: domain
    allowActions:
      - ds.newEntity
      - ds.newDict

  - resourceType: category
    allowActions:
      - cat.brainstorm
      - cat.generateSRS
      - cat.exportHTML
      - cat.sealAll
      - cat.importProject
      - cat.exportPackage

  - resourceType: group
    allowActions:
      - grp.analyze
      - grp.review
      - grp.write
      - grp.seal
      - grp.import
      - grp.export

  - resourceType: item
    allowActions:
      - itm.analyze
      - itm.review
      - itm.write
      - itm.seal
      - itm.import
      - itm.export

  - resourceType: dataStandard
    allowActions:
      - ds.newDomain
      - ds.newEntity
      - ds.newDict
      - ds.importFile
      - ds.reverseDb
      - ds.reverseDdl
      - ds.reverseCode
```

**执行时机**: `ActionRegistry.execute()` 在执行 action 前，校验 actionId 是否在 `permissions[context.resourceType].allowActions` 中。

**与 Tauri Permission 的关系**:
- `permissions.yml`: 业务级权限（"这个资源类型可以执行这些动作"）
- Tauri Capability: 系统级权限（"这个窗口可以调用这些 Rust command"）
- 两层独立校验，缺一不可

### 3.4 ContextMenuResolver semanticType 过滤（🔧 能力补全）

**问题**: 当前 `ContextMenuResolver.getContextMenuItems()` 只匹配 `resourceType`，不按 `semanticType` 进一步筛选。

**修复**: 在 context-menus.yml 的 `appliesTo` 中已支持 `semanticType` 字段，需要在 `ContextMenuResolver` 中增加过滤逻辑：

```typescript
export function getContextMenuItems(context: ContextObject): ContextMenuItem[] {
  const config = getConfig().contextMenus
  if (!indexCache) indexCache = buildIndex(config)

  const candidates = indexCache.get(context.resourceType) ?? []
  const items: ContextMenuItem[] = []

  for (const candidate of candidates) {
    // 按 semanticType 进一步筛选
    if (candidate.semanticType && candidate.semanticType !== context.semanticType) {
      continue
    }
    for (const item of candidate.items) {
      items.push(item)
    }
  }

  return items
}
```

`★ Insight ─────────────────────────────────────`
ContextMenuResolver 的索引结构需要从 `Map<resourceType, Menu[]>` 扩展为 `Map<resourceType, Map<semanticType?, Menu[]>>`，否则每次右键都要遍历所有候选项做 semanticType 比对。对于 30+ 菜单配置来说影响不大，但这是正确的数据结构思维——把过滤条件编码到索引键中。
`─────────────────────────────────────────────────`

### 3.5 Tauri 原生右键菜单（🆕 新能力）

**问题**: 当前右键菜单使用 Vue 渲染的 DOM 元素（`FileTreeContextMenu.vue`），在 Tauri 桌面环境下不是原生体验。PRD §9 明确建议使用 `@tauri-apps/api/menu` 的 `Menu.popup()`。

**设计**: 双轨制——桌面环境使用 Tauri 原生菜单，浏览器环境降级为 Vue DOM 菜单。

```typescript
// src/orchestration/ContextMenuPresenter.ts (新增)

import { Menu, MenuItem, PredefinedMenuItem } from '@tauri-apps/api/menu'
import { isTauri } from '../api/transport'

export async function showContextMenu(
  items: ContextMenuItem[],
  context: ContextObject,
  x: number,
  y: number,
): Promise<void> {
  if (isTauri()) {
    await showNativeContextMenu(items, context, x, y)
  } else {
    await showVueContextMenu(items, context, x, y)
  }
}
```

**Tauri 路径**:
```typescript
async function showNativeContextMenu(
  items: ContextMenuItem[],
  context: ContextObject,
  x: number,
  y: number,
) {
  const menuItems = []
  for (const item of items) {
    if (item.type === 'separator') {
      menuItems.push(await PredefinedMenuItem.new({ item: 'Separator' }))
      continue
    }

    const enabled = evaluateExpression(item.enabledWhen, getRuntimeState(), context)

    menuItems.push(await MenuItem.new({
      id: item.id,
      text: tt(item.label!),  // i18n key → display text
      enabled,
      action: async () => {
        const params = resolveParams(item.params, context)
        if (item.action) {
          await getActionRegistry().execute(item.action, params, getRuntime())
        } else if (item.flow) {
          await getFlowExecutor().execute(item.flow, { ...params })
        }
      },
    }))
  }

  const menu = await Menu.new({ items: menuItems })
  await menu.popup({ x, y })
}
```

**Vue 降级路径**: 保持现有 `FileTreeContextMenu.vue` 的数据驱动渲染方式。

### 3.6 动态 Provider 子菜单（🆕 新能力）

**问题**: menus.yml 中已经声明了 `type: submenu` + `provider: recentProjects`，但 MenuBar 渲染时未处理这种动态菜单。

**设计**:

```typescript
// src/orchestration/MenuProviderRegistry.ts (新增)

type MenuProvider = () => MenuItemConfig[]

const menuProviders = new Map<string, MenuProvider>()

export function registerMenuProvider(id: string, provider: MenuProvider) {
  menuProviders.set(id, provider)
}

export function resolveProviderMenu(id: string): MenuItemConfig[] {
  const provider = menuProviders.get(id)
  if (!provider) {
    console.warn(`[MenuProvider] Unknown provider: ${id}`)
    return []
  }
  return provider()
}
```

注册示例：
```typescript
registerMenuProvider('recentProjects', () => {
  const recentStore = useRecentProjectsStore()
  return recentStore.projects.map(p => ({
    id: `project.openRecent.${p.id}`,
    label: p.name,
    action: 'project.open',
    params: { path: p.path },
  }))
})
```

### 3.8 Tauri Capability 拆分（🔧 安全加固）

**当前状态**: `src-tauri/capabilities/default.json` 中所有权限集中在 main 窗口。

**建议拆分**:

```text
src-tauri/capabilities/
├── workbench.json    # 主工作台：项目管理、分析、编辑
```

## 五、安全模型（四层完整）

```
第 1 层 — visibleWhen/enabledWhen          = UX 控制    (YML 配置)
第 2 层 — permissions.yml allowActions      = 业务权限    (YML 配置) ← v2 新增
第 3 层 — ActionRegistry allowedCommands    = 应用安全    (TS 白名单)  ← v2 新增
第 4 层 — Tauri Capability + Permission     = 系统安全    (capabilities/) ← v2 增强
第 5 层 — Rust Command 参数校验              = 资源安全    (Rust 代码)
```

PRD 原文是四层，v2 细化为五层，将"业务权限"和"应用安全"拆开：permissions.yml 控制"谁能做什么"，allowedCommands 控制"哪些 backend 入口合法"。
