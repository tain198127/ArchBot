1. # 需求

   tauri 中，如何进行业务编排？比如，我想编排，菜单下面有哪些子菜单。menu 下面有哪些子 menu，右键选中哪个子菜单时，显示哪些右键菜单。点击哪个右键菜单，调用哪个事件或者哪个方法。这些我不想硬编码到代码中我想抽象到一套 yml 中，然后我的程序根据我的 yml 来执行。

2. # 结论

   可以做，而且这正是你这种 **Tauri 本地工具 / AI 工作台 / 分析型 IDE** 应该采用的方式。

   但要先纠正一个假设：

   > 你不能、也不应该把“所有方法本身”都配置化到 YML 中。
   > 你真正应当配置化的是：**菜单结构、展示条件、可用条件、动作标识、动作参数、动作链路**。
   > 真正具有执行能力的代码，必须由受控的 **Action Registry / Command Bus / Plugin Registry** 提供。

   否则，你的 YML 会逐渐演变成一个“可执行脚本语言”，最后出现三个问题：

   1. **安全问题**：配置文件可以直接调用文件读写、Shell、模型调用、删除资源等高风险能力。
   2. **可维护性问题**：方法重命名、参数变化、权限变化会使大量 YML 失效。
   3. **不可测试问题**：业务规则散落在配置表达式中，无法稳定做静态校验和单元测试。

   Tauri v2 已经原生支持多级菜单、菜单项动作回调，以及通过 `Menu.popup()` / `Submenu.popup()` 弹出上下文菜单；同时，它通过 Capabilities 与 Permissions 控制前端 WebView 能调用哪些后端能力。也就是说：**Tauri 提供菜单与权限基础设施，你需要在其上实现一层配置驱动的业务编排引擎。** ([Tauri](https://v2.tauri.app/learn/window-menu/)) ([Tauri](https://v2.tauri.app/reference/javascript/api/namespacemenu/)) ([Tauri](https://v2.tauri.app/security/capabilities/)) ([Tauri](https://v2.tauri.app/security/permissions/))

   ------

   # 一、你实际要抽象的内容

   你的需求不是简单的“菜单配置”，而是四类模型：

   | 模型           | 解决的问题                     | 示例                                        |
   | -------------- | ------------------------------ | ------------------------------------------- |
   | 菜单树模型     | 菜单下面有哪些子菜单           | 项目 → 分析 → 接口分析                      |
   | 上下文菜单模型 | 右键选中不同对象时显示什么菜单 | 选中 Java Controller 时出现“分析接口”       |
   | 条件模型       | 菜单是否显示、是否可点击       | 只有项目已加载并且文件类型为 Java 时可执行  |
   | 动作编排模型   | 点击后执行什么动作链           | 读取文件 → 调用分析器 → 保存结果 → 刷新视图 |

   因此，不要把设计目标局限为：

   ```yaml
   菜单 -> 方法
   ```

   应该设计为：

   ```yaml
   菜单树 / 上下文对象 / 条件规则 / 动作标识 / 动作编排 / 权限约束
   ```

   ------

   # 二、推荐总体架构

   ```mermaid
   flowchart LR
       A[menu.yml / context-menu.yml / action-flow.yml] --> B[配置加载器]
       B --> C[Schema 校验器]
       C --> D[菜单树解释器]
       C --> E[条件表达式解释器]
       C --> F[动作编排解释器]
   
       D --> G[Vue 页面菜单]
       D --> H[Tauri Native Menu]
       E --> I[根据当前选中对象过滤菜单]
       H --> J[用户点击菜单]
       G --> J
   
       J --> F
       F --> K[Action Registry]
       K --> L[前端动作]
       K --> M[Tauri invoke 后端命令]
       K --> N[插件动作]
       M --> O[Rust 受控能力层]
   
       P[Tauri Capabilities / Permissions] --> O
   ```

   ## 建议分层

   | 层           | 职责                                           | 是否配置化          |
   | ------------ | ---------------------------------------------- | ------------------- |
   | 展现层       | 左侧菜单、顶部菜单、右键菜单、快捷键           | 是                  |
   | 条件层       | 何时显示、何时禁用、对什么对象生效             | 是                  |
   | 编排层       | 多个 Action 的执行顺序、输入输出传递           | 是                  |
   | 动作注册层   | `project.open`、`analysis.api.scan` 等动作实现 | 半配置化，必须注册  |
   | 能力安全层   | 文件系统、进程执行、网络、模型调用、数据库访问 | 不应由 YML 任意开放 |
   | Tauri 后端层 | Rust command、权限与系统资源访问               | 代码控制            |

   # 三、配置文件建议拆分

   不要把所有东西塞进一个巨大的 `menu.yml`。建议拆成：

   ```text
   config/
   ├── menus.yml               # 顶部菜单 / 侧边菜单结构
   ├── context-menus.yml       # 右键菜单规则
   ├── actions.yml             # 动作定义
   ├── flows.yml               # 多步骤业务编排
   ├── predicates.yml          # 可复用条件定义
   └── permissions.yml         # 你自己的业务权限映射，不等价于 Tauri capability
   ```

   # 四、YML 模型设计

   ## 1. 顶部菜单或导航菜单：`menus.yml`

   ```yaml
   version: "1.0"
   
   menus:
     - id: project
       label: 项目
       icon: folder
       children:
         - id: project.open
           label: 打开项目
           shortcut: CmdOrCtrl+O
           action: project.open
   
         - id: project.recent
           label: 最近项目
           type: submenu
           provider: recentProjectsProvider
   
     - id: analysis
       label: 分析
       children:
         - id: analysis.interface
           label: 接口分析
           children:
             - id: analysis.interface.scan
               label: 扫描 Controller 接口
               action: analysis.controller.scan
               visibleWhen: project.loaded == true
               enabledWhen: project.language == "java"
   
             - id: analysis.interface.export
               label: 导出接口清单
               action: analysis.interface.export
               enabledWhen: analysis.interfaceResult.exists == true
   
         - id: analysis.callChain
           label: 调用链分析
           action: analysis.callChain.run
           visibleWhen: project.loaded == true
   ```

   这个文件只负责：

   - 菜单层级；
   - 显示名称；
   - 图标；
   - 快捷键；
   - 对应的动作标识；
   - 简单可见性与可用性规则。

   它不负责真正的代码执行。

   ------

   ## 2. 右键菜单：`context-menus.yml`

   你真正关心的是：

   > 我右键选中了什么对象，这个对象有哪些允许操作。

   因此右键菜单不要绑定到页面，而应绑定到 **上下文对象类型**。

   ```yaml
   version: "1.0"
   
   contextMenus:
     - id: java-controller-context
       appliesTo:
         resourceType: file
         fileExtension: java
         semanticType: controller
   
       items:
         - id: controller.analyzeApi
           label: 分析该 Controller 的接口
           action: analysis.controller.single
           params:
             filePath: "${context.resource.path}"
   
         - id: controller.traceService
           label: 分析调用链
           action: analysis.callChain.fromController
           params:
             filePath: "${context.resource.path}"
   
         - type: separator
   
         - id: controller.openSource
           label: 打开源码
           action: editor.openFile
           params:
             filePath: "${context.resource.path}"
   
     - id: table-context
       appliesTo:
         resourceType: databaseTable
   
       items:
         - id: table.findUsage
           label: 查找涉及该表的接口
           action: analysis.table.findUsage
           params:
             tableName: "${context.resource.tableName}"
   
         - id: table.viewFlow
           label: 查看数据流向
           action: analysis.dataFlow.byTable
           params:
             tableName: "${context.resource.tableName}"
   ```

   ## 上下文对象建议统一结构

   无论用户右键的是文件、接口、表、流程节点还是测试用例，都包装成统一的 `ContextObject`：

   ```ts
   interface ContextObject {
     resourceType: 'file' | 'directory' | 'api' | 'databaseTable' | 'flowNode' | 'testCase';
     resource: {
       id?: string;
       name: string;
       path?: string;
       tableName?: string;
       apiPath?: string;
     };
     semanticType?: 'controller' | 'service' | 'mapper' | 'vuePage' | 'sql' | 'config';
     attributes?: Record<string, unknown>;
   }
   ```

   这样，右键菜单系统完全不依赖某个具体页面。

   # 五、动作模型：不要在 YML 里直接写方法名

   ## 错误设计

   ```yaml
   action:
     class: com.xxx.AnalysisService
     method: scanController
   ```

   或者：

   ```yaml
   action:
     javascript: "window.service.analysis.scan(context.path)"
   ```

   这类设计短期看起来灵活，长期必然失控。

   ## 正确设计：动作标识 + 参数

   ```yaml
   actions:
     - id: analysis.controller.scan
       label: 扫描全部 Controller
       executor: backend
       command: analysis_scan_controllers
       input:
         projectPath: "${state.project.path}"
       result:
         storeAs: analysis.interfaceResult
         notify: true
   
     - id: editor.openFile
       label: 打开文件
       executor: frontend
       handler: editor.openFile
       input:
         filePath: "${context.resource.path}"
   ```

   YML 中只声明：

   - 动作叫什么；
   - 需要什么输入；
   - 结果放到哪里；
   - 是否刷新视图；
   - 是否弹提示；
   - 是否需要确认；
   - 是否需要权限。

   真正执行动作的实现，由代码注册。

   # 六、Action Registry：整个方案的核心

   你的程序启动时注册一批稳定动作：

   ```ts
   type ActionHandler = (
     params: Record<string, unknown>,
     runtime: ActionRuntime
   ) => Promise<unknown>;
   
   class ActionRegistry {
     private handlers = new Map<string, ActionHandler>();
   
     register(actionId: string, handler: ActionHandler) {
       if (this.handlers.has(actionId)) {
         throw new Error(`Duplicate action: ${actionId}`);
       }
       this.handlers.set(actionId, handler);
     }
   
     async execute(
       actionId: string,
       params: Record<string, unknown>,
       runtime: ActionRuntime
     ) {
       const handler = this.handlers.get(actionId);
   
       if (!handler) {
         throw new Error(`Unknown action: ${actionId}`);
       }
   
       return handler(params, runtime);
     }
   }
   ```

   注册动作：

   ```ts
   import { invoke } from '@tauri-apps/api/core';
   
   const registry = new ActionRegistry();
   
   registry.register('project.open', async () => {
     return await invoke('open_project');
   });
   
   registry.register('analysis.controller.scan', async (params) => {
     return await invoke('analysis_scan_controllers', {
       projectPath: params.projectPath
     });
   });
   
   registry.register('analysis.callChain.fromController', async (params) => {
     return await invoke('analysis_call_chain_from_controller', {
       filePath: params.filePath
     });
   });
   
   registry.register('editor.openFile', async (params, runtime) => {
     runtime.editor.open(String(params.filePath));
   });
   ```

   这里仍然有代码，但这不是“硬编码菜单逻辑”，而是建立稳定的业务能力目录：

   ```text
   YML 决定何时、在哪里、按照什么顺序调用能力
   代码只负责提供受控能力
   ```

   这才是可维护的配置驱动架构。

   # 七、多步骤业务编排：`flows.yml`

   当你不仅希望“点击菜单调用一个方法”，而是希望：

   > 右键 Controller → 扫描接口 → 分析 Service → 追踪 Mapper → 识别表 → 导出报告 → 刷新页面

   就需要 `flow` 模型。

   ```yaml
   version: "1.0"
   
   flows:
     - id: analysis.fullControllerTrace
       label: 完整分析 Controller 调用链
   
       input:
         filePath: "${context.resource.path}"
   
       steps:
         - id: validateFile
           action: validation.ensureJavaController
           params:
             filePath: "${input.filePath}"
   
         - id: parseController
           action: analysis.controller.single
           params:
             filePath: "${input.filePath}"
           outputAs: controllerResult
   
         - id: traceServices
           action: analysis.service.trace
           params:
             controllerResult: "${steps.controllerResult}"
           outputAs: serviceResult
   
         - id: traceMappers
           action: analysis.mapper.trace
           params:
             serviceResult: "${steps.serviceResult}"
           outputAs: mapperResult
   
         - id: identifyTables
           action: analysis.database.identifyTables
           params:
             mapperResult: "${steps.mapperResult}"
           outputAs: tableResult
   
         - id: saveGraph
           action: workspace.saveAnalysisResult
           params:
             type: callChain
             controller: "${steps.controllerResult}"
             services: "${steps.serviceResult}"
             mappers: "${steps.mapperResult}"
             tables: "${steps.tableResult}"
   
         - id: refreshView
           action: ui.refreshPanel
           params:
             panelId: call-chain-panel
   ```

   右键菜单只需要引用这个流程：

   ```yaml
   - id: controller.fullTrace
     label: 完整分析调用链
     flow: analysis.fullControllerTrace
     params:
       filePath: "${context.resource.path}"
   ```

   # 八、条件规则：不要让 YML 变成一门复杂编程语言

   例如：

   ```yaml
   visibleWhen: project.loaded == true
   enabledWhen: context.semanticType == "controller"
   ```

   这类条件可接受。

   但不要逐渐支持：

   ```yaml
   enabledWhen: >
     project.loaded == true &&
     user.permission.contains("analysis") &&
     context.resource.path.endsWith(".java") &&
     results.filter(x => x.type === "controller").length > 0
   ```

   这种写法一旦泛滥，配置就变成了不可测试脚本。

   ## 建议采用两级条件体系

   ### 简单表达式

   ```yaml
   visibleWhen: project.loaded == true
   enabledWhen: context.semanticType == "controller"
   ```

   ### 复杂条件注册为 Predicate

   ```yaml
   enabledWhen:
     predicate: canAnalyzeSelectedController
   ```

   代码注册：

   ```ts
   predicateRegistry.register('canAnalyzeSelectedController', (state, context) => {
     return state.project.loaded === true &&
            context.resourceType === 'file' &&
            context.semanticType === 'controller' &&
            context.resource.path?.endsWith('.java') === true;
   });
   ```

   这样复杂规则可以被单元测试覆盖，而不是埋在配置中。

   # 九、Tauri 中如何创建原生菜单与右键菜单

   Tauri v2 的 JavaScript 菜单 API 支持：

   - `Menu`
   - `Submenu`
   - `MenuItem`
   - `CheckMenuItem`
   - `IconMenuItem`
   - `Menu.popup()` 作为右键上下文菜单
   - 菜单项的 `action` 回调

   官方文档中，`MenuOptions.items` 可以包含菜单项或子菜单；`MenuItemOptions.action` 用于响应菜单点击；`Menu.popup()` 可将菜单弹出为指定窗口上的上下文菜单。([Tauri](https://v2.tauri.app/reference/javascript/api/namespacemenu/)) ([Tauri](https://v2.tauri.app/reference/javascript/api/namespacemenu/)) ([Tauri](https://v2.tauri.app/reference/javascript/api/namespacemenu/))

   ## 根据 YML 构建右键菜单

   ```ts
   import { Menu, MenuItem, PredefinedMenuItem } from '@tauri-apps/api/menu';
   
   async function showContextMenu(context: ContextObject, x: number, y: number) {
     const matchedConfig = contextMenuConfigs.find(config =>
       matchContext(config.appliesTo, context)
     );
   
     if (!matchedConfig) {
       return;
     }
   
     const items = [];
   
     for (const item of matchedConfig.items) {
       if (item.type === 'separator') {
         items.push(await PredefinedMenuItem.new({ item: 'Separator' }));
         continue;
       }
   
       if (!evaluateVisible(item.visibleWhen, context)) {
         continue;
       }
   
       items.push(await MenuItem.new({
         id: item.id,
         text: item.label,
         enabled: evaluateEnabled(item.enabledWhen, context),
         action: async () => {
           const params = resolveParams(item.params, context);
           await executeConfiguredAction(item, params);
         }
       }));
     }
   
     const menu = await Menu.new({ items });
   
     await menu.popup({
       x,
       y
     });
   }
   ```

   Vue 组件中监听右键：

   ```vue
   <template>
     <div
       class="tree-node"
       @contextmenu.prevent="openContextMenu"
     >
       {{ node.name }}
     </div>
   </template>
   
   <script setup lang="ts">
   async function openContextMenu(event: MouseEvent) {
     const context = buildContextObjectFromNode(props.node);
   
     await showContextMenu(
       context,
       event.clientX,
       event.clientY
     );
   }
   </script>
   ```

   # 十、Tauri 后端只暴露受控 Command

   例如你的系统需要扫描 Java 项目、读取文件、生成分析报告。

   Rust 后端暴露固定能力：

   ```rust
   #[tauri::command]
   async fn analysis_scan_controllers(project_path: String) -> Result<String, String> {
       // 校验路径
       // 扫描 Controller
       // 返回结构化结果
       Ok("result".to_string())
   }
   
   #[tauri::command]
   async fn analysis_call_chain_from_controller(file_path: String) -> Result<String, String> {
       // 分析 Controller -> Service -> Mapper -> SQL
       Ok("call-chain-result".to_string())
   }
   ```

   前端配置只能引用你允许的动作：

   ```yaml
   actions:
     - id: analysis.controller.scan
       executor: backend
       command: analysis_scan_controllers
   ```

   但这里还需要再加一层限制：

   ```ts
   const allowedBackendCommands = new Set([
     'analysis_scan_controllers',
     'analysis_call_chain_from_controller',
     'open_project'
   ]);
   
   function validateConfiguredCommand(command: string) {
     if (!allowedBackendCommands.has(command)) {
       throw new Error(`Command is not allowed: ${command}`);
     }
   }
   ```

   **不要让 YML 中任意写一个 command 名称就直接调用 `invoke()`。**

   # 十一、Tauri 权限层如何处理

   Tauri v2 中，前端 WebView 能否访问某个能力，不是默认全部开放，而是由 **Capabilities** 与 **Permissions** 控制：

   - Capability 绑定某些窗口或 WebView；
   - Permission 决定哪些 command 或 plugin 能被调用；
   - 权限文件位于 `src-tauri/capabilities` 与 `src-tauri/permissions`；
   - 多个 capability 绑定同一个窗口时，其权限边界会合并，应谨慎设计。([Tauri](https://v2.tauri.app/security/capabilities/)) ([Tauri](https://v2.tauri.app/security/permissions/))

   例如主窗口需要菜单能力和事件能力：

   ```json
   {
     "$schema": "../gen/schemas/desktop-schema.json",
     "identifier": "main-capability",
     "description": "Main workbench window permissions",
     "windows": ["main"],
     "permissions": [
       "core:menu:default",
       "core:event:default",
       "core:window:default"
     ]
   }
   ```

   对于你自己的分析命令，应继续细分：

   ```text
   工作区窗口：
   - 可读取当前项目
   - 可触发分析
   - 可写入分析报告
   
   设置窗口：
   - 只能修改配置
   - 不允许执行扫描、脚本或文件系统写入
   
   插件管理窗口：
   - 只能安装、启停插件
   - 不允许直接访问用户项目文件
   ```

   这对你的产品很重要，因为未来它不仅会“读本地文件”，还可能“写文件、执行命令、调用大模型、运行测试”。配置驱动越强，权限边界越要严格。

   ------

   # 十二、针对你的产品，建议采用的业务对象模型

   你不应把菜单只绑定到“文件”。你的系统未来会分析：

   - 项目；
   - 页面；
   - Controller；
   - Service；
   - Mapper；
   - SQL；
   - 表；
   - 接口；
   - 调用链；
   - 业务规则；
   - 测试用例；
   - Agent 任务。

   因此，建议统一抽象为：

   ```yaml
   resourceTypes:
     - project
     - module
     - page
     - controller
     - service
     - mapper
     - sql
     - table
     - api
     - callChain
     - businessRule
     - testCase
     - agentTask
   ```

   然后配置右键菜单：

   ```yaml
   contextMenus:
     - id: api-context
       appliesTo:
         resourceType: api
   
       items:
         - id: api.traceImplementation
           label: 追踪实现代码
           flow: analysis.api.traceImplementation
   
         - id: api.generateTest
           label: 生成接口测试
           flow: testing.api.generateCase
   
         - id: api.findRules
           label: 查找业务规则
           flow: analysis.api.extractRules
   
     - id: business-rule-context
       appliesTo:
         resourceType: businessRule
   
       items:
         - id: rule.findSource
           label: 定位来源代码
           action: analysis.rule.locateSource
   
         - id: rule.validate
           label: 验证规则完整性
           flow: validation.rule.check
   ```

   这会比“文件右键菜单”更适合作为你的产品底层模型。

   # 十五、你需要避免的三个设计陷阱

   ## 1. 不要把 YML 直接设计成可调用任意方法

   错误方向：

   ```yaml
   method: invoke
   args:
     command: "${userInput}"
   ```

   这等于给配置文件开放了后端执行入口。

   应改成：

   ```yaml
   action: analysis.controller.scan
   ```

   并由动作注册表映射到受控实现。

   ------

   ## 2. 不要把权限判断只放在菜单显隐中

   菜单隐藏不等于权限控制。

   即使某个菜单不显示，恶意前端逻辑仍可能直接调用后端命令。因此：

   ```text
   菜单 visibleWhen      = 用户体验控制
   Action Registry 校验  = 应用执行控制
   Tauri Permission      = 系统安全控制
   Rust Command 校验     = 最终资源控制
   ```

   四层缺一不可。

   ------

   ## 3. 不要一开始就追求“全动态插件化”

   你的第一版不应该同时实现：

   - 动态菜单；
   - 动态流程；
   - 动态插件安装；
   - 动态权限；
   - 动态脚本；
   - 动态 Agent；
   - 动态模型编排。

   这会直接把复杂度打爆。

   ## 第一阶段应只实现

   | 能力                      | 是否做 |
   | ------------------------- | ------ |
   | 菜单树 YML 配置           | 做     |
   | 右键菜单 YML 配置         | 做     |
   | Action Registry           | 做     |
   | 简单 Flow 串行编排        | 做     |
   | Predicate 注册            | 做     |
   | 参数解析 `${context.xxx}` | 做     |
   | 任意 JS / Rust 脚本执行   | 不做   |
   | 第三方插件热加载          | 暂不做 |
   | 可视化流程设计器          | 暂不做 |

   ------

   # 十七、第一版最小可行实现

   第一版只需要四个配置对象：

   ```yaml
   actions:     # 原子动作映射
   flows:       # 多步骤执行链
   ```

   以及五个运行时模块：

   ```text
   ConfigLoader

   FlowExecutor
   PermissionGuard
   ```

   第一批动作可以限定为：

   ```text
   project.open
   project.refresh
   editor.openFile
   analysis.controller.single
   analysis.controller.scan
   analysis.callChain.run
   analysis.table.findUsage
   report.exportCsv
   workspace.saveResult
   ui.refreshPanel
   ```

   这已经能够支撑你最核心的场景：

   ```text
   右键 Controller
     → 分析接口
     → 追踪 Service / Mapper / SQL / 表
     → 生成结构化结果
     → 展示调用链
     → 导出分析报告
   ```

   # 最关键的判断

   你现在不是在设计一个“可配置菜单系统”，而是在设计一个：

   > **以资源对象为中心、以动作注册为安全边界、以 YML 为编排描述语言的桌面业务执行框架。**

   菜单只是它的入口之一。未来入口还会包括：

   - 命令面板；
   - Agent 自动任务；
   - 工作流节点；
   - 快捷键；
   - 文件变化事件；
   - 用户自然语言指令；
   - 批量分析任务。

   因此底层千万不要写成：

   ```text
   菜单点击 -> 调方法
   ```

   而要写成：

   ```text
   触发器 -> 上下文对象 -> 规则匹配 -> Action / Flow -> 受控执行能力 -> 状态结果
   ```

   这套抽象才有机会支撑你后续的 AI IDE、代码分析、规则抽取、测试生成和多 Agent 协作。
